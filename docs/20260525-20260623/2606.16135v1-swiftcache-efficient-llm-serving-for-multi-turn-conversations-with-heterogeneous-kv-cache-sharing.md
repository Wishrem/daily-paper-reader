## 摘要
多轮对话是LLM应用中的基本场景，广泛应用于聊天机器人和AI代理。随着对话的进行，历史token不断累积。现有系统缓存这些token的键值（KV）对以避免重复计算。然而，有限的GPU内存（HBM）容量常常迫使这些KV缓存被卸载到CPU内存或SSD，随着上下文增长，KV缓存重新加载的延迟成本越来越高。同时，受限的HBM容量也限制了最大推理长度，从而限制了对话中可支持的轮数。为了解决这两个挑战，我们提出了SwiftCache，一个协作推理系统，使得异构模型能够共享服务器内未充分利用的GPU内存和NVLink带宽。具体来说，KV缓存需求低的模型将空闲的GPU内存捐赠出来，用于存储高需求模型的前缀缓存，通过NVLink实现跨模型KV缓存共享，避免缓慢的PCIe传输。SwiftCache进一步通过仅在本地GPU内存中保留当前活动层的KV缓存来减少内存压力，从而实现更长上下文的推理。我们在真实工作负载上的实验表明，与vLLM和SGLang相比，SwiftCache将P99首次token生成时间（TTFT）降低了最多69%，并将最大上下文长度延长了最多3.98倍，同时对共存模型的干扰最小。

## Abstract
Multi-turn conversation is a fundamental scenario in LLM applications, widely used in chatbots and AI agents. As the conversation evolves, historical tokens accumulate continuously. Existing systems cache their key-value (KV) pairs to avoid redundant computation. However, limited GPU memory (HBM) capacity often forces these KV caches to be offloaded to CPU memory or SSD, making KV cache reloads increasingly costly in terms of latency as the context grows. Meanwhile, the constrained HBM capacity also limits the maximum inference length, thereby restricting the number of turns that can be supported in a conversation.   To address these two challenges, we propose SwiftCache, a collaborative inference system that enables heterogeneous models to share underutilized GPU memory and NVLink bandwidth within a server. Specifically, models with low KV cache demand donate idle GPU memory to store the prefix cache of high-demand models, allowing cross-model KV cache sharing over NVLink and avoiding slow PCIe transfers. SwiftCache further reduces memory pressure by keeping only the KV cache of the currently active layer in local GPU memory, thereby enabling longer-context inference. Our experiments on real-world workloads show that SwiftCache reduces P99 time-to-first-token (TTFT) by up to 69% and extends maximum context length by up to 3.98x compared to vLLM and SGLang, with minimal interference to co-located models.

# SwiftCache 论文总结

## 1. 核心问题与整体含义（研究动机与背景）

- **核心问题**：多轮对话场景下，LLM 服务面临两个关键挑战：
  - **KV 缓存传输延迟高**：历史 token 累积导致 KV 缓存占用大量 GPU 内存，超出 HBM 容量后必须卸载到 CPU 内存或 SSD 并通过 PCIe 加载。PCIe 传输成为主要瓶颈（实验中传输时间占比高达 86.2%）。
  - **最大上下文长度受限**：HBM 容量限制了模型实际能处理的最大序列长度，例如 LWM-1M-Text 在 32GB V100 GPU 上只能处理 18K tokens，即使 96GB H20 也只能处理 160K tokens，远低于其宣称的 1M。
- **整体含义**：现有系统（如 vLLM、SGLang）采用层级缓存但未充分利用同服务器内其他模型的空闲 GPU 内存和高速 NVLink 带宽。SwiftCache 提出通过异构模型协作共享资源，以低延迟、低干扰的方式改善多轮对话的推理效率。

## 2. 方法论

### 核心思想
- **跨模型 KV 缓存共享**：利用同一服务器内不同模型（master 与 worker）的闲置 GPU 内存和 NVLink 带宽。低 KV 缓存需求模型（worker）将空闲缓存空间捐赠给高需求模型（master），通过 NVLink 直接传输，避免 PCIe 瓶颈。
- **层流缓存（Layer Stream Cache, LSC）**：推理时仅保留当前活跃层的 KV 缓存于本地 GPU，其他层的 KV 缓存存储在 worker GPU 上，按需流式加载，降低本地内存占用，延长上下文长度。
- **弹性缓存（Elastic Cache）**：采用块主（block-major）布局的 KV 缓存组织方式，支持 O(1) 时间复杂度的动态扩容/缩容，适应工作负载变化。

### 关键技术细节
1. **Layer Stream Cache（LSC）**：
   - 计算 LSC 容量：`N_LSC = min(∑K_i, K_master)`，其中 `K_i` 为第 i 个 worker 能存储的 master 全层块数，`K_master` 为 master 本地能存储的单层块数。
   - 剩余本地内存用于 Regular Cache（RC），存储全层块，提供最低访问延迟。
   - 推理时，master 在执行某层注意力前，通过 NVLink 异步预取该层 KV 缓存到 LSC，同时将计算后的新 KV 缓存写回 worker。传输与计算可重叠（layer-wise pipelining）。

2. **弹性缓存与块主布局**：
   - 传统层主布局（`n_layers x n_blocks`）扩容/缩容需 O(n_layers × n_blocks) 数据移动。
   - 块主布局：`n_blocks x n_layers`，同一索引的块跨层连续存储，扩容/缩容仅需在末尾追加或释放，O(1) 复杂度。
   - 为保证内存对齐，使用最小弹性单元（MEU）—— master 和 worker 块大小的最小公倍数（LCM）确定 scaling 单位。

3. **Master-Worker 协调**：
   - Worker 根据当前请求长度和历史窗口最大需求，调用 ScaleUp/ScaleDown 算法动态调整分配块数，并与 master 同步块表（block table）。
   - ScaleUp: 若当前块数不足，则按 MEU 倍数增加；ScaleDown: 定期检查过去 t 秒内最大请求，若当前块数超出需求，按 MEU 倍数释放。

4. **传输计算重叠**：
   - 第一层：预取可在调度后立即开始；后续层：等待前一层注意力完成且其 KV 传输结束后开始异步加载，与 FFN 计算重叠，隐藏传输延迟。

## 6. 主要结论与发现

- **P99 TTFT 降低**：相比 SGLang (HiCache) 降低 67%~69%，相比 vLLM (LMCache) 降低 54%~58%。
- **最大上下文长度提升**：在 96GB H20 上，Qwen3-32B 提升至 2.78×~6.66×；LWM 提升至 1.81×~2.22×。在低显存 GPU (24GB) 上提升更显著，最高达 6.15×。
- **干扰极小**：worker 模型 TTFT 增加 ≤9.7%，TPOT 增加 ≤6.5%。主要原因是 KV 传输仅消耗网络和少量内存带宽，不占用计算资源。
- **传输开销可忽略**：即使 Qwen3-32B 模型，KV load/store 时间占比 <1.4%，且可通过计算重叠降至 <0.1%。
- **LSC 与块主布局有效性**：LSC 将单层缓存容量扩大 L 倍（L 为层数），弹性缓存支持 O(1) 缩放，内存对齐机制避免了碎片。

## 7. 优点

- **创新性**：首次提出利用同服务器不同模型间的 NVLink 带宽和空闲 GPU 内存进行 KV 缓存共享，突破了传统仅依赖本机内存的局限。
- **实用性**：兼容现有主流推理优化（Continuous Batching, PagedAttention, FlashAttention），易于集成；对共存模型影响微小，适合多模型部署场景。
- **效率高**：块主布局和弹性缩放设计精巧，实现 O(1) 重分配；层感知流水线重叠有效隐藏传输延迟。
- **实验全面**：覆盖多种模型、多种显存容量、多种工作负载，对比了多个 SOTA 基线，数据扎实。

## 8. 不足与局限

- **应用范围有限**：目前仅支持单服务器多 GPU 环境，无法扩展到跨服务器集群（因 NVLink 不能跨服务器直连），限制了在大规模分布式部署中的适用性。
- **未考虑精度影响**：未讨论 KV 压缩或稀疏方法可能带来的精度损失，但 SwiftCache 本身不进行有损压缩，故无精度损失。
- **实验偏差风险**：
  - 仅测试了 Qwen3 系列和 LWM 模型，未覆盖其他架构（如 MoE 模型）或更大参数规模的模型，通用性需进一步验证。
  - 干扰实验仅测量了 worker 的 TTFT/TPOT 比率，未分析更高请求率下 worker 是否会因频繁缩放而出现性能抖动。
- **缺少微基准测试**：未单独评测弹性缓存缩放的实际延迟、块表同步开销、LCU 对齐机制对内存利用率的影响。
- **未讨论 GPU 内存碎片问题**：虽然块主布局避免了层间碎片，但动态分配可能仍产生内部碎片，尤其当 master/worker 块大小不成整数倍时。
- **长上下文场景局限性**：对于超过千亿参数的模型，LSC 存储单层 KV 缓存可能仍较大，对 worker GPU 内存依赖较高。
