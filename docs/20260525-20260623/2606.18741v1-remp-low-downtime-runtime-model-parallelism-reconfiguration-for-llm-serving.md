## 摘要
当前的大语言模型（LLM）推理系统普遍采用张量并行（TP）和流水线并行（PP）的组合来部署超大规模模型。然而，现有系统将模型并行拓扑视为静态配置，无法在运行时灵活调整。这种刚性设计与现实场景中动态变化的推理工作负载产生了根本性矛盾。最先进的系统缺乏在线重配置能力，只能通过重启服务来切换配置，导致数分钟的服务中断、KV缓存丢失以及高昂的重新计算开销。为解决这一问题，本文提出ReMP，一种支持低停机时间的运行时模型并行重配置框架。ReMP通过三项关键技术实现动态调整：（1）将模型并行拓扑与运行时状态解耦，避免完全重建服务；（2）设计二维KV缓存迁移机制，在TP/PP变化后保留可复用的缓存状态；（3）实现端到端的在线重配置。实验表明，ReMP能够在参数规模从7B到70B的模型上，在1-7秒内完成大多数拓扑切换，与重启方法相比实现了数十倍到数百倍的加速。此外，ReMP在动态工作负载下显著优于固定配置，在TTFT、TPOT和输出吞吐量方面展现出更优性能。

## Abstract
Current large language model (LLM) inference systems universally deploy ultra-large-scale models using a combination of Tensor Parallelism (TP) and Pipeline Parallelism (PP). However, existing systems treat the model parallelism topology as a static configuration that cannot be flexibly adjusted at runtime. This rigid design creates a fundamental contradiction with the dynamically changing inference workloads in real-world scenarios. State-of-the-art systems lack online reconfiguration capabilities and can only switch configurations by restarting the service, resulting in several minutes of service interruption, KV cache loss, and prohibitive recomputation overhead. To address this problem, this paper presents ReMP, a runtime model parallelism reconfiguration framework that supports low downtime. ReMP achieves dynamic adjustment through three key techniques: (1) decoupling the model parallelism topology from runtime state to avoid full service reconstruction; (2) designing a two-dimensional KV cache migration mechanism to preserve reusable cache states after TP/PP changes; and (3) implementing end-to-end online reconfiguration. Experiments demonstrate that ReMP can complete most topology switches within 1-7 seconds on models ranging from 7B to 70B parameters, achieving speedups of tens to over a hundred times compared to the restart approach. Moreover, ReMP significantly outperforms fixed configurations under dynamic workloads, delivering superior performance in terms of TTFT, TPOT, and output throughput.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：当前大语言模型（LLM）推理服务普遍采用张量并行（TP）和流水线并行（PP）的组合来部署大规模模型，但这些并行拓扑在服务启动时即固定，无法在运行时调整。实际生产环境中，LLM推理流量在不同时间段存在显著差异（如白天高峰、夜间低谷），而最优的TP/PP配置会随负载变化（低负载时需低延迟→高TP；高负载时需高吞吐→高PP）。静态拓扑导致系统无法适应动态工作负载。
- **现有方案局限**：最先进的系统（如vLLM）不支持在线重配置，只能通过重启服务来切换拓扑，导致分钟级停机、KV缓存全部丢失、重新进行昂贵的预填充计算，切换代价极高，使得动态调整在实践中不可行。
- **本文目标**：提出ReMP，一个低停机时间的运行时模型并行重配置框架，使LLM服务能够根据实时工作负载灵活切换TP/PP配置，无需完全重启服务。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- 将模型并行拓扑与运行时状态（模型权重、KV缓存、通信组、工作进程、调度器元数据）解耦，使得拓扑切换时仅需更新受影响的组件，而非重建整个服务实例。

### 关键技术细节

1. **共享模型权重存储（Shared Weight Store）**  
   - 服务启动时将完整模型状态字典加载到CPU共享内存中，所有工作进程共享。  
   - 拓扑切换时，工作进程直接从共享状态字典中提取目标分片（根据目标TP/PP确定层范围和张量切片），无需重新从磁盘加载检查点。

2. **二维KV缓存迁移（Two-Dimensional KV Cache Migration）**  
   - PP决定层归属（哪个流水线阶段拥有某层的KV缓存），TP决定KV头归属（同一层内不同头分布在TP各rank上）。  
   - 拓扑切换时，ReMP构建迁移计划：对于每个目标rank，计算其应拥有的层（PP）和头范围（TP），然后与旧rank拥有的头范围求交集，非空交集即需通过本地拷贝或P2P通信迁移的KV切片。  
   - 采用逐层流式迁移（layer-wise streaming），每次只迁移一层，降低峰值内存占用。  
   - 若目标容量小于当前活跃缓存占用，调度器会抢占部分请求并回收其缓存块。

3. **MPU状态空间（MPU State Space）**  
   - 为多个候选拓扑预构建并行状态快照（包括TP组、PP组、rank映射等），切换时直接应用目标快照，避免动态重建NCCL通信组的开销。

4. **工作进程生命周期管理（Worker Lifecycle Management）**  
   - 工作进程分为活跃（active）和待机（standby）状态，拓扑切换时根据目标world size，将多余worker置为standby或唤醒standby worker，避免进程创建/销毁代价。

5. **调度器适配（Scheduler Adaptation）**  
   - 切换前冻结调度器，切换后更新KV缓存配置（层数、块数、张量形状等），调整块管理器，必要时抢占请求以释放缓存。

### 重配置事务流程（文字描述）
- 输入：旧拓扑、新拓扑、调度器状态、KV缓存元数据、共享模型状态、预构建MPU状态空间。
- 阶段：服务于旧拓扑 → 静默调度器 → 准备worker集（scale-down时先迁移KV再置为standby；scale-up时先唤醒再迁移） → 应用目标MPU状态 → 并发执行模型分片重载（从CPU共享内存）与KV缓存迁移 → 更新调度器元数据 → 恢复服务。
- 通过重载与迁移的重叠，将切换关键路径时间从 `T_worker + T_mpu + T_kv + T_model + T_sched` 优化为 `T_worker + T_mpu + max(T_kv, T_model) + T_sched`。

4. **可适应性**：最优拓扑随模型架构和请求压力变化，ReMP能通过快速切换抓住性能优势，证明模型并行拓扑可作为运行时可调资源。

## 7. 优点

- **首创性**：首次实现LLM推理服务中的运行时TP/PP联合重配置，突破静态拓扑限制。

- **二维KV迁移**：同时处理PP的层维度和TP的头维度迁移，采用流式迁移控制内存峰值，设计清晰。
- **端到端集成**：在vLLM v1中深度实现，与executor、worker管理、KV缓存管理器、通信初始化等模块无缝结合，具备实际部署基础。
- **实验全面**：覆盖多个模型规模、架构、硬件平台，且消融实验验证了重叠优化效果。

## 8. 不足与局限

- **候选拓扑需预知**：MPU状态空间要求候选拓扑集合在服务启动前已知，不能支持任意动态拓扑（如运行时完全随机切换）。论文提到支持任意拓扑需额外动态组缓存机制，未实现。
- **容量约束下的缓存丢失**：当目标拓扑的KV容量小于当前活跃缓存时，部分请求会被抢占并重新计算，无法完全避免丢失，可能影响部分用户体验。
- **实验范围局限**：仅测试了BurstGPT轨迹，未在更长序列或更复杂真实流量（如混合短长请求）下评估；未与异构执行（如预填充与解码分离）场景结合。
- **资源假设**：依赖充足的主机内存存储完整模型状态（2TB/960GB），在内存受限节点可能不适用。
- **仅考虑同构GPU**：假设所有GPU型号相同，未考虑异构集群（如不同代GPU混合）下的重配置。
- **无与其他动态框架对比**：虽然当前无同类系统，但未来可能出现类似工作，当前缺少横向对比（如PipeLive仅支持PP重配置）。
