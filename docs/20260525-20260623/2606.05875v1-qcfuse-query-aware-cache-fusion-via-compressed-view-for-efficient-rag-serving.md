## 摘要
检索增强生成通过将生成过程锚定在外部证据上来提升大语言模型的回答质量，但处理检索到的上下文使得预填充阶段成为主要的服务成本。RAG缓存融合通过复用检索块的预计算键值缓存，并在当前提示下选择性重新计算词元来降低这一成本。然而，现有选择器在质量和效率之间面临两难：快速的查询无关或最终层查询到上下文选择器可能遗漏与请求相关的证据，而全视图查询感知选择器在重新计算之前需要广泛的上下文和层可见性，从而阻碍了逐层缓存融合流水线。我们提出了QCFuse，一种用于RAG缓存融合的压缩视图查询感知选择器。QCFuse使用块锚点查询探测，将用户查询状态条件化在紧凑的逐块锚点上，并通过关键层剖析在不经过全层检查的情况下识别重新计算词元。我们在SGLang中实现了QCFuse，并在六个数据集上对四个开放权重的大语言模型进行了评估。QCFuse达到了全预填充级别的质量。在相同质量下，与完全预填充相比，QCFuse平均预填充速度提升1.7倍，相比最强质量保持基线ProphetKV提升1.5倍。

## Abstract
Retrieval-augmented generation (RAG) improves large language model (LLM) answer quality by grounding generation in external evidence, but processing retrieved contexts makes the prefill stage a dominant serving cost. RAG cache fusion reduces this cost by reusing precomputed key-value (KV) caches for retrieved chunks and selectively recomputing tokens under the current prompt. Existing selectors, however, face a dilemma between quality and efficiency: fast query-agnostic or final-layer query-to-context selectors can miss request-relevant evidence, whereas full-view query-aware selectors require broad context and layer visibility before recomputation and therefore stall the layer-wise cache-fusion pipeline. We present QCFuse, a compressed-view query-aware selector for RAG cache fusion. QCFuse uses chunk-anchor query probing to condition user-query states on compact per-chunk anchors and critical-layer profiling to identify recomputation tokens without all-layer inspection. We implement QCFuse in SGLang and evaluate it on four open-weight LLMs across six datasets. QCFuse reaches full-prefill-level quality. At matched quality, QCFuse achieves an average prefill-time speedup of 1.7x over full prefill and 1.5x over ProphetKV, the strongest quality-preserving baseline.

# QCFuse: 查询感知缓存融合的高效RAG服务（中文总结）

## 1. 核心问题与整体含义

检索增强生成（RAG）通过将外部证据注入大语言模型来提升回答质量，但其预填充阶段（prefill）因需处理大量检索文字而成为主要服务瓶颈。KV缓存融合（cache fusion）通过复用预计算的块级KV缓存并选择性重算关键token来加速预填充，但现有选择器在**质量与效率**间存在两难：

- **查询无关**或**仅用最终层**的选择器速度快，但易遗漏与当前查询相关的证据；
- **全视图查询感知**选择器（如ProphetKV）质量高，然而需要在重算前加载完整的上下文和全层KV，导致流水线阻塞，抵消了融合加速优势。

**QCFuse**提出**压缩视图查询感知选择器**，通过块锚点查询探测和关键层剖析，在保持查询感知质量的同时避免全视图加载，实现高效流水线并行。

## 2. 方法论

### 核心思想
QCFuse将选择器需要观察的“证据视图”沿两个维度压缩：
- **Token视图压缩**：每个可复用块内仅保留少量代表性锚点（anchor token），用户查询仅在锚点缓存上做条件化，而非整个上下文。
- **层视图压缩**：离线分析模型各层定位证据的能力，仅选择少数关键层（critical layers）进行注意力评分，避免全层扫描。

### 关键技术细节
- **离线阶段**：
  1. 预计算每个块的KV缓存（PIC），并基于KVzip规则从每个块内选择10%的锚点token（根据自监督重建探针评分）。
  2. 对模型进行关键层剖析：使用锚点缓存+全层注意力与黄金答案位置（来自SQuAD/NewsQA/NQ）的Recall@10%来评估每层定位能力，选Top-3层作为关键层集合L★。
- **在线阶段**：
  1. 加载检索块的锚点KV缓存，拼接成压缩上下文C_A；同时预取关键层的上下文键。
  2. 用户查询在锚点缓存上计算注意力得分（仅关键层），得到锚点条件化的查询状态。
  3. 使用这些状态与关键层原始位置键计算上下文token重要性分数，选择Top-ρ个token进行重算。
- **流水线集成**：选择完成后启动逐层稀疏重算，与后续层KV加载重叠。

公式示意（文字说明）：
- 重要性分数 = sum_{l in L★} Attn_l( query states conditioned on anchor KV , keys of original context at layer l )
- 最终选择集P = TopK( 重要性分数, floor(ρ*N) )，其中ρ为重算比例。

## 3. 实验设计

### 数据集与基准
- **数据集**：
  - **LongBench**：MuSiQue（多跳）、2WikiMQA、HotpotQA（多跳QA）
  - **RULER**：多查询检索（mq）、多值检索（mv）、变量跟踪（vt）
  - 每个数据集200个样本，上下文由20个512-token块组成。
- **评估指标**：
  - 质量：F1（LongBench）、字符串匹配SM（RULER），均归一化。
  - 延迟：首token时间（TTFT），包括选择、缓存加载、重算和首token生成。
  - 吞吐量：每秒请求数。

### 对比方法
- **Full prefill**（全预填充，质量上界）
- **Direct PIC reuse**（直接复用，速度上界）
- **CacheBlend**（查询无关，浅层KV偏差）
- **EPIC**（查询无关，块边界固定重算）
- **FusionRAG**（仅最终层查询感知）
- **ProphetKV**（全视图查询感知）

所有缓存融合方法使用相同的检索块、重算比例、缓存放置和逐层流水线，仅选择策略不同。

## 6. 主要结论与发现

- QCFuse达到**全预填充级别的质量**（与ProphetKV相当，且在某些任务上超过Full prefill）。
- 在匹配质量下，QCFuse比Full prefill**平均预填充加速1.7×**，比ProphetKV**平均加速1.5×**。
- 在长上下文下，QCFuse质量稳定优于查询无关和弱查询感知基线，接近全视图方法。
- 低带宽场景下，QCFuse的选择阶段耗时远小于ProphetKV，避免成为I/O瓶颈。
- 在高请求负载下，QCFuse保持较低TTFT，吞吐量曲线右移。
- 锚点保留10%即可捕获大部分质量增益；Top-3关键层接近全层效果。

## 7. 优点

- **创新性**：首次将压缩视图（token+层双维压缩）引入缓存融合选择器，解决了查询感知与流水线兼容的矛盾。
- **实用性**：只需离线每个块预计算少量锚点和一次模型级关键层剖析，在线开销极小；可集成进主流推理引擎（SGLang）。
- **实验全面**：覆盖多个模型家族（Mistral、Llama、Qwen）、多种任务、多种baseline，并进行了大量消融。
- **结果显著**：在保持质量的前提下实现1.5-1.7倍加速，且加速效果在苛刻条件（低带宽、高并发）下依然稳健。

## 8. 不足与局限

- **实验规模局限**：仅评估了7B-14B的开源模型，未在更大模型（如70B、MoE等）或闭源模型上验证；样本数200 per dataset偏小，可能掩盖长尾行为。
- **离线剖析依赖**：关键层选择和锚点规则（KVzip）需要黄金答案位置标签（SQuAD等），对特定模型/任务可能需要重新profile；论文仅在四个模型上profile一次，未测试跨领域迁移。
- **部署复杂性**：锚点存储和关键层集需要额外的离线管理；虽然开销小，但增加了系统维护成本。
- **应用限制**：假设推理引擎可访问Transformer的KV缓存和层间状态，不适用于黑盒API服务；仅适用于可调用内部结构的开源LLM。
- **未剖析统计显著性**：缺少误差线和多次运行方差报告，结论的稳健性可通过重复实验进一步验证。
