Abstract 

As the input length of large language model (LLM) serving continues to grow, the KV cache has become a dominant bot-tleneck in AI infrastructure. It limits GPU memory capacity, serving concurrency, cache reuse, and distributed scalability. Several important problems, including position-independent KV cache, prefix KV cache compression, hot/cold KV cache separation, and distributed KV cache management, all depend on how the KV cache is represented and managed. How-ever , existing serving systems largely rely on a monolithic KV cache abstraction, where the KV cache is treated as a homogeneous sequence of token-level memory blocks and managed with similar policies across attention heads and serv-ing scenarios. We observe that KV cache utility is highly structured across KV heads: different heads exhibit different functional roles, attention distances, and runtime importance. Therefore, a full KV cache is not always necessary for every head, token range, or serving scenario. We present RedKnot, a head-aware KV cache management system for LLM serving. RedKnot breaks the conventional monolithic KV cache abstraction by decomposing the KV cache along KV heads, whose importance and effective at-tention ranges vary significantly across serving scenarios .This head-level decomposition turns the KV cache from a monolithic tensor abstraction into a structured memory object, enabling RedKnot to uniformly support position-independent KV reuse, prefix KV compression, hot/cold KV separation, and distributed KV placement while preserving output fidelity and improving resource efficiency, without requiring model retraining or fine-tuning. RedKnot establishes a new foun-dation for AI infrastructure by transforming the KV cache from a monolithic, passive runtime artifact into a dynamic, model-aware runtime substrate for scalable LLM serving.

2 Background 

In this section, we review two lines of work that motivate our design. First, position-independent KV-cache reuse enables systems to amortize prefill computation across prompts even when reusable text chunks do not appear as exact prefixes. Second, structured sparsity in attention and FFNs shows that different model components may require different amounts of context and computation during inference. Together, these observations suggest that KV-cache management should be both chunk-aware and structure-aware, rather than treating all tokens, layers, and heads uniformly. 

2.1 Position-Independent KV Cache 

Many emerging workloads, such as retrieval-augmented gen-eration (RAG), long-context question answering, few-shot learning, and agentic applications, repeatedly use the same text chunks, but these chunks do not always appear as ex-act prefixes. This motivates position-independent KV cache (PIC), where the system attempts to reuse precomputed KV cache even when the reused text appears at different positions in the final prompt. CacheBlend [47] is a representative work in this direction. It observes that reusable text chunks in RAG workloads may appear in non-prefix positions, where their precomputed KV caches cannot be directly reused because they miss cross-attention with the preceding texts. It addresses this problem by blending precomputed KV caches and selectively recom-2puting a subset of tokens to recover output quality. EPIC [15] further formalizes position-independent caching and intro-duces a serving system that enables modular KV cache reuse regardless of token chunk positions. It mitigates the attention-sink effect introduced by independently cached chunks and improves TTFT and throughput with negligible or no accu-racy loss. ProphetKV [41] focuses on long-context RAG and improves selective recomputation by prioritizing tokens according to their semantic relevance to the user query, ad-dressing the crowding-out effect where globally salient but query-irrelevant tokens consume the limited recomputation budget. In parallel, CacheSlide [24] studies non-prefix KV cache reuse in agentic workloads, where previously generated contexts can be reused across repeated execution paths. Its reuse model, however, assumes a relatively fixed order among reusable KV states. This assumption differs from typical PIC settings, such as RAG and multi-agent applications, where reusable chunks are often retrieved, reordered, and composed dynamically. Therefore, CacheSlide addresses an important but different point in the design space of non-prefix KV cache reuse. 

2.2 Structured Sparsity in Attention and FFNs 

Modern LLM inference contains two major computational components: grouped-query attention (GQA) [2, 8, 16, 34, 38] and feed-forward networks (FFNs) [26, 36, 40]. Recent stud-ies show that both components exhibit strong runtime sparsity, suggesting that dense computation is not always necessary for every head, token, or neuron. 

Multi-head attention sparsity. Long-context attention ex-hibits strong and relatively stable head-level heterogeneity. Existing studies suggest that whether an attention head be-haves as a global/retrieval head or a local/streaming head is largely a model-intrinsic property associated with its layer and head index, rather than being determined from scratch for each input context. In other words, once a specific (layer , head ) is identified, its effective attention scope can often be profiled offline and reused across requests. StreamingLLM [44] iden-tifies the attention sink phenomenon, where several initial tokens receive high attention even when they are not semanti-cally important. By preserving these sink tokens and recent tokens, StreamingLLM supports stable streaming generation with a bounded KV cache. DuoAttention [43] further sepa-rates attention heads into retrieval heads and streaming heads: retrieval heads require full-context access, while streaming heads mainly attend to recent tokens and attention sinks and can use a constant-length KV cache. RazorAttention [37] similarly observes that most heads primarily focus on local context, while only a small number of retrieval heads need access to the full KV cache. MInference [17] shows that long-context attention heads follow distinct sparse patterns, such as A-shape, Vertical-Slash, and Block-Sparse patterns, and assigns different sparse computation strategies to different heads during prefill. These works suggest that attention computation is naturally structured across heads. More importantly, this structure can be represented as a stable per-head cache requirement. For example, a profiled local head at a given layer may only need a bounded window of recent tokens plus a small number of sink tokens, while a profiled global head may require full-context KV access. Therefore, different heads may require different context ranges and KV cache residency policies, and such policies can be determined at the granularity of (layer , head )

rather than recomputed from scratch for every request. 

FFN sparsity. In parallel, FFN computation also exhibits strong activation sparsity. DejaVu [26] observes contextual sparsity in Transformer inference, where only a small input-dependent subset of attention heads and MLP parameters is needed to approximate the dense model output. Pow-erInfer [36] exploits the power-law distribution of neuron activations in LLM inference, separating frequently acti-vated hot neurons from cold neurons to reduce GPU mem-ory demand and CPU-GPU data movement. More recent activation-sparsity methods, such as CATS [20], TEAL [23], and ProSparse [35], further show that FFN or activation spar-sity can be induced or exploited in modern LLMs to reduce computation with limited quality degradation. Together, these studies show that LLM inference contains structured sparsity beyond token-level reuse. For multi-head attention, KV-cache heads can be largely summarized as global heads and local heads: global heads require long-range KV access, while local heads mainly rely on recent tokens, attention sinks, or bounded windows, enabling substantial cache reduction with negligible accuracy loss. FFNs fur-ther expose activation-level sparsity, where only a subset of neurons or channels dominates computation for a given input. 

3 Motivation and Opportunity 

In this section, we analyze why existing position-independent KV cache (PIC) systems leave most of their promised speedup unrealized, and we identify the structural opportunities that motivate RedKnot’s design. 

3.1 Limitations of Token-Level Recovery 

Existing PIC systems, such as CacheBlend, EPIC, and ProphetKV, attempt to exploit sparsity of attention [9, 17, 37, 43] by selecting a subset of tokens for recomputation or correction. The underlying assumption is that only a small number of tokens are critical for recovering the quality gap between reused KV cache and full prefill. However, this token-level sparsity becomes less effective under multi-head attention. Different heads may attend to different token sub-sets and exhibit different position sensitivity. As a result, the tokens that need correction for one head may differ from those required by another head. When the system must recover all 3Figure 2: For short contexts, prefill TTFT is dominated by FFN computation rather than KV-cache construction. 

heads of a selected token together, the effective recomputa-tion set becomes the union of head-specific important tokens. This union can cover a large portion of the chunk, forcing the system to recompute many tokens to recover accuracy. This creates a fundamental limitation for token-level PIC recovery. 

Even if each individual head is sparse, their important token sets may be diverse across heads. Token-level meth-ods cannot exploit this per-head sparsity because they collapse all heads of a token into one recovery decision. 

Therefore, they face an unfavorable trade-off: selecting fewer tokens may leave some head-specific errors uncorrected and hurt output quality, while selecting more tokens improves fidelity but quickly reduces the TTFT benefit of KV reuse. Beyond the attention-level bottleneck, existing PIC sys-tems also overlook a complementary challenge that arises in short-context scenarios. For short input sequences, the feed-forward network (FFN) computation dominates the time-to-first-token (TTFT) rather than attention. As shown in Figure 2, FFN layers consistently dominate TTFT across both Qwen3-32B and Llama-3.3-70B, accounting for over 57% of total prefill time at short contexts (2K–8K tokens). As con-text length increases, the FFN share gradually declines while the attention share rises, reflecting the quadratic scaling of attention with sequence length. Notably, even at 32K tokens, FFN still contributes 44.4% and 53.4% of prefill TTFT for Qwen3-32B and Llama-3.3-70B, respectively, underscoring FFN computation as the primary bottleneck for long-context RAG prefill workloads. In this regime, even a perfect KV cache reuse strategy that eliminates all attention recomputa-tion yields only marginal TTFT reduction, because the FFN cost remains unaddressed. As a result, conventional PIC implementations may still suffer from substantial TTFT in short-context settings, which limits the extension of PIC from RAG-centric workloads to broader application domains, such as the widely adopted agent-based scenarios. Reducing the FFN computation cost can substantially improve the gen-erality of PIC systems, enabling their benefits to extend beyond long-context RAG workloads .

We observe that some heads change signifi-cantly because their attention is strongly affected by the pre-ceding prefix, while others remain nearly unchanged because their attention is mostly confined to local context. This result clearly shows that only a small fraction of KV heads are global or prefix-sensitive, while the majority exhibit local and prefix-robust behavior. This stability allows the system to profile the property of each KV head offline and reuse the profiling result across re-quests. When the same text chunk is reused after a different prefix, the system can therefore focus recovery computation on the small set of global heads that are sensitive to prefix changes, while directly reusing most local-head KV cache. This provides a finer-grained recovery path than token-level recomputation and reduces unnecessary attention computa-tion. 

3.3 Opportunity for Efficient PIC 

The two limitations identified above jointly point to a two-dimensional opportunity for building an efficient PIC system 4that preserves output quality while substantially reducing prefill TTFT. 

From token-level to head-level recovery. Section 3.1 shows that token-level recomputation cannot exploit per-head spar-sity, because forcing all heads of a selected token to be re-covered together inflates the recomputation set to the union of head-specific important tokens. Section 3.2 further shows that only a small fraction of heads (12.5–15.6% across rep-resentative models) are prefix-sensitive and require recovery at all, while the vast majority of local heads can be reused verbatim with negligible quality loss. These two findings to-gether motivate shifting the recovery granularity from tokens 

to heads : the system recomputes only the global heads across the entire chunk, leaving local-head KV cache untouched. Be-cause global/local assignments are stable across requests, they are determined once via offline profiling and applied at zero per-request classification overhead. Head-level recovery thus simultaneously achieves higher fidelity (every prefix-sensitive head is fully refreshed) and lower cost (no redundant work on the ∼85% of local heads) than token-level alternatives, es-caping the unfavorable accuracy–latency trade-off described in Section 3.1. 

Reducing FFN Cost via Token-Selective Recovery. Sec-tion 3.1 also shows that FFN computation contributes sig-nificantly to prefill TTFT across all context lengths, and is the overwhelming bottleneck in short-context, multi-agent workloads where attention-side savings are inherently limited. (Section 2.2) shows that FFN computation contributes signif-icantly to prefill TTFT. RedKnot reduces this cost through token-selective FFN recovery. After head-aware attention recovery, RedKnot estimates token importance from the re-covered attention signal. Important token states execute the original dense FFN, while other token states bypass the FFN update through the residual path. This optimization is inde-pendent of KV-cache sparsity and can accelerate short-context workloads where attention-side savings are limited. This spar-sity is structurally independent of the KV cache and can be exploited in parallel with head-level KV reuse. By evaluating only the predicted active neurons per token, the system skips the majority of FFN multiply-accumulate operations without altering the attention computation path. Critically, FFN spar-sity is effective regardless of context length, making it the primary lever for accelerating short-context scenarios where KV reuse alone cannot deliver meaningful TTFT reduction. 

3.4 Challenges for Efficient PIC 

Although the sparsity identified in Sections 3.1 and 3.2 pro-vides a clear opportunity, translating it into real TTFT reduc-tion, lower computational cost, and high-accuracy outputs raises two practical challenges. 

C1: Discrete per-head sparsity undermines token-level recovery. Although attention sparsity is widely recog-nized [17, 37, 43], the discrete and heterogeneous nature of per-head sparsity makes it fundamentally difficult to exploit at the token level. Under multi-head attention, different heads attend to different subsets of tokens, and their sparse patterns vary substantially across heads. When a token-level PIC sys-tem selects a fixed set of tokens for recomputation, it must take the union of the important token sets across all heads in order to satisfy every head. This union quickly expands to cover a large fraction of the chunk, forcing the system to re-compute nearly as many tokens as a full prefill, which defeats the purpose of KV cache reuse. Conversely, if the system selects only a small token subset to limit recomputation cost, a different failure mode arises. For any head whose important tokens are not included in the selected set, the attention scores computed during correction will be based on an incomplete context, leaving the error in that head’s KV cache unaddressed and degrading output quality. Furthermore, even for tokens that are selected for recomputation, the correction remains incomplete. Consider token i chosen as an important token shared across all heads: because tokens 0 to i − 1 still use the stale reused KV cache rather than freshly computed values, the attention context seen by token i during recomputation is itself corrupted. As a result, the recomputed KV of token i

cannot fully recover to its ground-truth full-prefill value, and this error propagates forward through the sequence. This cas-cading inaccuracy is an inherent limitation of any token-level PIC scheme: reusing KV entries for preceding tokens while selectively recomputing later ones introduces an irrecoverable dependency on stale context. 

C2: HBM bandwidth becomes the bottleneck under long-context PIC. PIC is primarily motivated by long-context RAG, where retrieved passages are concatenated into prompts spanning tens to hundreds of thousands of tokens. In this set-ting, KV-cache traffic becomes a dominant cost because the cache size scales linearly with sequence length and propor-tionally with the number of layers and attention heads [30]. For a 70B-class model at 128K context length, the KV cache alone can exceed 40 GB, consuming a large fraction of GPU HBM [1]. Modern GPUs make this bottleneck more pro-nounced: compute throughput has grown much faster than HBM bandwidth. The compute–memory balance point is al-ready about ∼156 FLOP/Byte on A100 and ∼295 FLOP/Byte on H100 SXM [5, 27]. Long-context LLM inference, espe-cially decode and sparse prefill, often falls below this thresh-old and is therefore limited by memory traffic rather than arithmetic throughput [1, 19, 30, 50]. 

4 RedKnot Design 

In this section, we introduce RedKnot, a system that acceler-ates long-context LLM inference with two key components: (i) Elastic Sparsity, which exploits head-level KV and FFN sparsity to reduce redundant attention computation with mini-mal accuracy loss; and (ii) SegPagedAttention, a head-aware KV-cache layout that avoids loading irrelevant head data and improves HBM bandwidth utilization and throughput. 5Figure 4: Overview of RedKnot. 

4.1 Overview of RedKnot 

RedKnot consists of two core components: (i) an Elastic Sparsity module, and (ii) a module that stores data at the gran-ularity of KV-cache heads. We next describe the end-to-end workflow of RedKnot, highlighting how these modules in-teract during inference. Layer-wise elastic recovery. Elastic Sparsity adopts a layer-wise recovery strategy. In shallow layers, Elastic Sparsity uses local attention recovery with full FFN computation. This conservative design preserves the early residual stream, where hidden states are more sensitive to perturbations and errors can be amplified by subsequent layers. In deep layers, Elas-tic Sparsity enables global-head attention recovery together with sparse FFN computation. Since deeper layers exhibit stronger semantic selectivity and more concentrated attention behavior, Elastic Sparsity recomputes prefix-sensitive global heads, reuses prefix-robust local heads, and applies FFN only to important tokens. 

Head-aware attention recovery. For global heads, Elastic Sparsity recomputes their KV states under the new prefix, unless the reused chunk is already covered by standard pre-fix reuse. This conservative policy preserves long-range de-pendency modeling and prevents prefix-induced errors from propagating through retrieval-oriented heads. For local heads, Elastic Sparsity reuses most cached KV states because their effective attention range is bounded. Specifically, for a local head at token i, Elastic Sparsity only recomputes the local visible set 

W (i) = Ssink ∪ [max (0, i − w), i],

where w is the local window size and S ∗ sink denotes the reserved sink-token positions. For example, when i = 1000 and w = 256 , Elastic Sparsity recomputes the sink tokens and the local window around positions 744 to 1000 , while directly reusing the remaining local-head KV cache. This policy preserves local continuity while avoiding unnecessary recomputation over distant tokens that are invisible to local heads. 

Partial sparse FFN recovery. Head-aware attention recovery reduces attention-side recovery cost, but FFN computation can still remain on the critical path of prefill. Therefore, after the recovered attention states are produced in deep layers, Elastic Sparsity applies partial sparse FFN recovery. Elastic Sparsity uses the recovered attention signal to estimate token importance. Tokens with high importance execute the dense FFN, while other tokens follow the residual identity path. In this way, Elastic Sparsity spends FFN computation only where correction is likely to affect the final hidden states. The two sparsity dimensions are complementary. Head-aware attention recovery reduces unnecessary KV recomputa-tion across heads, while partial sparse FFN recovery avoids dense FFN replay across tokens. Together, they form an elastic sparsity mechanism that adapts recovery cost to the correction demand of each layer and reused chunk. 4.3 SegPagedAttention 

To address Challenge 2 discussed in Section 3.4, we de-sign an module named SegPagedAttentionto achieve high concurrency.SegPagedAttention is the storage and execution substrate that supports Elastic Sparsity’s head-aware recovery policy. Existing paged KV-cache systems usually organize cached states at token-block granularity, where all KV heads within the same token block are allocated, transferred, and ac-cessed together. This abstraction is efficient for conventional prefix caching, but it is too coarse-grained for head-aware PIC recovery. In Elastic Sparsity, different heads may follow different runtime policies: global-style heads always require full-range attention over all mapped KV pages, while local-style heads mostly need only sink tokens and recent-window pages. Therefore, the KV cache should expose the head dimension to the runtime. To this end, SegPagedAttention introduces a head-segmented KV layout. Instead of treating a token block as the only management unit, SegPagedAtten-tion represents cached states as head segments . Each head segment is indexed by (ℓ, h, s), where ℓ is the layer id, h is the KV head id, and s is the segment id within that head stream. A head segment contains a contiguous range of KV states for one specific KV head, and is mapped to one or more consecu-tive virtual pages in the paged KV-cache space. For example, a segment (L0, h0, s0) can map to pages 3– 4 , while another segment (L0, h6, s1) can map to pages 5 –7 . This mapping preserves the benefits of paged KV management, such as vir-tualized allocation, page-table lookup, and page-level transfer, while making each KV head independently addressable. The key advantage of this layout is that it decouples head-specific execution policies from token-block storage .With token-level paging, if any head inside a token block requires full-context recovery, the system tends to load or recompute the entire block across all heads. This collapses head-specific sparsity into a coarse token-level decision. In contrast, SegPagedAttention allows the runtime to access only the pages associated with the required head segment .A global-style segment can access its full mapped page range and execute full-range attention except when it happens to be in the prefix, while a local-style segment can access only the pages that overlap with its visible region, such as sink pages and recent-window pages. Thus, different head seg-ments can follow different attention execution policies with-out forcing all heads in the same token range to be processed together. SegPagedAttention is especially useful for position-independent KV reuse. When a cached chunk is reused after a new prefix, Elastic Sparsity recomputes or repairs only the prefix-sensitive head segments, while directly reusing the prefix-robust local segments whenever their visible pages are unchanged. This avoids the token-level union problem dis-cussed in Section 3.1: even if different heads require different correction regions, their sparse structures can be preserved separately at the head-segment level. 5.2 Quality and TTFT Comparison 

We first evaluate the accuracy of RedKnot and comprehen-sively assess its performance from multiple perspectives. Fig-ure 7 compares RedKnot with two representative token-level PIC baselines, CacheBlend and ProphetKV, along both an-swer quality and end-to-end prefill latency. We report four quality dimensions, including exact match (EM), token-level F1, first-token top-1 match, and first-token top-10 match, to-gether with TTFT speedup over the dense prefill baseline. Overall, RedKnot achieves a more favorable quality–latency trade-off than token-level PIC baselines: it consistently deliv-ers higher answer fidelity while also achieving stronger TTFT reduction, especially at longer contexts. 

Quality comparison. From Figure 7 (b) and Figure 7 (c), RedKnot is consistently competitive in token-level F1 and clearly stronger in exact-match accuracy. On the two Llama-3.3-70B HotpotQA cases, RedKnot improves EM from 0 .60 to 0.80 , while CacheBlend and ProphetKV remain at or below the dense baseline. On the Qwen3-32B MultiFieldQA cases, RedKnot preserves the same EM as Dense, whereas token-level baselines collapse to nearly zero EM. Even in cases where ProphetKV or CacheBlend remain partially competi-tive in F1, their EM is generally lower, indicating that token-level sparse recomputation often fails to recover the exact answer even when surface token overlap remains nontrivial. The top- 1 and top-10 match statistics in Figure 7(d) further support this conclusion: RedKnot remains much closer to the dense prefill path in next-token prediction behavior, while token-level baselines exhibit noticeably larger divergence. The main reason is that RedKnot performs head-aware recovery instead of token-level recovery. Existing PIC base-lines such as CacheBlend and ProphetKV decide which tokens 

should be recomputed or corrected. However, under multi-head attention, different heads often attend to different token subsets and exhibit different sensitivity to prefix changes. As a result, token-level methods either recompute too few tokens, which leaves some head-specific errors uncorrected, or recom-pute too many tokens, which weakens the latency benefit. In contrast, RedKnot separates global heads from local heads. Prefix-sensitive global heads are explicitly recomputed, while prefix-robust local heads are largely reused with lightweight repair. This preserves head-specific sparse structure instead of collapsing all heads into a single token-level decision. Con-sequently, RedKnot is able to recover the dominant attention behavior more faithfully, which directly improves EM, F1, and next-token agreement. 

TTFT comparison. Figure 7(a) shows that RedKnot also achieves the best TTFT speedup in most long-context set-tings. Its advantage becomes more pronounced as the context grows. For example, on Llama-3.3-70B with 64K context, RedKnot reaches the largest speedup among all compared methods. This trend is consistent with the design goal of Red-Knot: reducing not only attention-side recovery cost, but also the remaining FFN bottleneck. This speed advantage comes from two complementary sources. First, RedKnot reduces attention recovery cost through head-aware execution. Only the small set of prefix-sensitive global heads are recomputed with full-range attention, while most local heads directly reuse cached KV states or only access bounded visible regions. This avoids the token-level union effect that forces CacheBlend and ProphetKV to reprocess a larger set of tokens than any single head actually requires. Second, RedKnot further accelerates prefill through sparse FFN recovery . Existing token-level PIC baselines mainly optimize the attention path, but still keep FFN computation dense. As context grows, this dense FFN increasingly dominates TTFT, limiting the speedup ceiling of token-level methods. RedKnot breaks this ceiling by applying sparse FFN computation only to important token states after attention recovery. Therefore, it reduces both attention-side and FFN-side computation, which explains why its TTFT 10 Figure 7: End-to-end comparison across latency, answer quality, and KV matching metrics. RedKnot consistently improves TTFT while preserving higher accuracy and stronger Top-K KV matching than position-independent cache baselines. 

scaling is better than that of CacheBlend and ProphetKV. 

Why RedKnot achieves a better quality–latency trade-off. The key difference is that RedKnot aligns its recovery granularity with the intrinsic structure of LLM inference. At the attention level, it exploits head-level heterogeneity: global heads are treated differently from local heads. At the FFN level, it exploits token-selective recovery, avoiding dense FFN replay for less important token states. By contrast, token-level PIC baselines operate at a coarser granularity and only partially exploit attention sparsity, while leaving FFN com-putation largely untouched. As a result, they suffer from a less favorable trade-off: aggressively reducing recomputation tends to hurt accuracy, while preserving accuracy requires more token recomputation and limits TTFT improvement. 

Summarize. Figure 7 shows that RedKnot achieves both 

higher quality and lower TTFT than representative token-level PIC baselines. Its quality advantage comes from head-aware recovery, which more accurately restores prefix-sensitive at-tention behavior. Its latency advantage comes from combin-ing head-aware attention recovery with sparse FFN recovery, thereby reducing both major components of long-context pre-fill. These results validate the central design principle of RedKnot: position-independent KV reuse should be recov-ered at the granularity of heads and important token states ,rather than only at token granularity.

5.5 System-Level Benefits Beyond Prefill 

After measuring answer quality, TTFT, and prefill FLOPs, we evaluate whether RedKnot also improves the system-level bottlenecks that appear in long-context serving: KV transfer under prefill–decode (PD) disaggregation, burst-mode request throughput, and concurrent session capacity. We run the PD experiments on the same 8 ×H800 testbed using Qwen3-32B and Llama-3.3-70B will vllm 0.13.0 without implementing segPagedAttention in order to compare with PagedAttention. The prefill and decode pools are placed on separate GPU groups, and the system transfers the produced KV cache from the prefill side to the decode side before first-token genera-tion. For throughput, we submit bursts of N simultaneous RAG requests and report completed requests per second. For capacity, we compute the number of concurrent sessions that fit in GPU memory under dense vLLM-style KV storage ver-sus RedKnot with SegPagedAttention, where per-head KV sparsity is physically materialized. 

KV transfer savings. Figure 13(a) shows that head-class sparsity directly reduces the KV bytes that must cross the PD boundary. On Llama-70B, RedKnot reduces transferred KV volume by 4.3 × at 8K and 5.7 × at 16K; transfer time drops by 4.1 × in both settings. On Qwen3-32B, the byte reduction is even larger, 5.6–6.3 × from 8K to 24K, while transfer-time speedup is 1.0–1.5 ×. The difference comes from the absolute size of the transfer. Llama-70B moves larger KV tensors, so reducing bytes removes real link work and converts almost linearly into wall-clock savings. Qwen3-32B has smaller per-request transfers; after compression, the remaining time is dominated more by launch, packing, and link setup overheads, so byte savings do not fully translate into transfer-time sav-ings. The reason the byte reduction is stable across workloads is structural. RedKnot’s head-class plan partitions KV by head: global heads keep the full context, local heads keep only the sink/recent window needed by their attention pattern, and retrieval-style heads keep only selected tokens. Dense 14 PD serving must ship the full [B, H, L, D] KV tensor even for heads that will never read most positions. RedKnot ships only the KV that each head can actually consume, turning per-head sparsity into a smaller network payload. 

Burst-mode throughput. Figure 13(b) reports the end-to-end impact under bursty concurrent load. With the cur-rent RedKnot backend, throughput improves from 0.14 to 0.20 req/s on Llama-70B at 8K ( +43% ), from 0.08 to 0.10 req/s at 16K ( +27% ), from 0.20 to 0.26 req/s on Qwen3-32B at 8K ( +28% ), from 0.11 to 0.13 req/s at 16K ( +19% ), and by 15% at Qwen3-32B 24K. These gains are smaller than the isolated TTFT and FLOP reductions because burst throughput includes more than prefill compute: request scheduling, KV packing, transfer, decode, and runtime or-chestration all remain in the critical path. The decreasing gain at longer contexts also exposes the limitation of the cur-rent backend. In this measurement, RedKnot still expresses per-head sparsity through a dense KV layout plus an attention mask. That path saves transfer bytes and some compute, but it still pays the SDPA mask penalty discussed in section 5.4. As context length grows, the dense+mask attention cost absorbs a larger fraction of the benefit. This is why the burst-throughput gain is positive but modest: the algorithm has removed un-necessary KV, while the backend has not yet turned all of that reduction into mask-free attention execution. 

Concurrent capacity per GPU. Figure 13(c) shows the benefit once the same sparsity is materialized by SegPagedAt-tention. Dense vLLM-style serving is often memory-bound rather than compute-bound: once GPU memory is filled by per-session KV cache, additional requests cannot be admitted even if compute units are available. With physical per-head KV storage, RedKnot increases the number of concurrent ses-sions per GPU from 4 to 31 at 32K context, a 7.8 × increase, and from roughly 3 to 14 at 64K context, a 4.7 × increase. Under a conservative pipelined serving model, this capac-ity expansion projects to 3.4–3.9 × higher capacity-bound throughput. This capacity result is the system-level conse-quence of SegPagedAttention. The current dense+mask im-plementation can decide which tokens a head should ignore, but the dense KV tensor still occupies memory for all heads and all positions. SegPagedAttention makes the sparse lay-out physical: local heads allocate pages only for their short windows, while global heads allocate full-context pages. For long-context RAG workloads, this changes the bottleneck from “how many full dense KV caches fit in HBM” to “how many per-head compact KV caches fit in HBM,” which is why the capacity gain is larger than the single-request throughput gain. 

Summary. RedKnot’s benefits extend beyond faster prefill. Head-class sparsity reduces PD transfer bytes by 4.3–6.3 ×,current burst-mode serving improves throughput by 15–43%, and SegPagedAttention converts the same sparsity into 4.7– 7.8 × more concurrent sessions per GPU. The three panels in fig. 13 therefore separate what the current implementation already realizes from what the physical per-head KV layout unlocks once integrated into the serving engine. 

6 Future Work 6.1 Engine Designed Around Per-Head Atten-tion 

The results in above point to a broader systems lesson: long-context serving engines should treat the attention head, not the token or the layer, as the basic unit of KV-cache management. Existing engines such as vLLM and SGLang were designed around dense PagedAttention-style tensors, where each layer stores a rectangular [B, H, L, D] KV block and every head is assumed to carry the same context length. This abstraction is simple and efficient for dense prefix caching, but it is mis-matched to the sparsity observed in RAG workloads. In Red-Knot, only a small fraction of heads need full-context access, while most heads are local and consume only a short sink/re-cent window. A layer-level cache abstraction therefore forces the system to pay the global-head cost for every head. This mismatch appears in three places. First, token-level PIC re-covery must select a token for all heads at once. Because different heads are sensitive to different positions, token-level recovery takes the union of head-specific important tokens and quickly approaches dense recomputation. Head-class recov-ery avoids this union effect by making the recovery decision at the (layer , head ) granularity: global heads are recomputed or restored with full context, while local heads reuse their cached KV within a bounded window. Second, dense KV storage hides the same sparsity from the kernel. Even if the algorithm knows that a local head needs only W ≪ L tokens, a dense [B, H, L, D] layout plus an attn_mask still presents a full-length tensor to SDPA, causing the FlashAttention path to be disabled and producing the 4.9–7.6 × mask penalty mea-sured in fig. 9. Third, PD disaggregation transfers KV at the tensor level unless the transport layer understands head classes. This is why per-head KV partitioning reduces trans-ferred bytes by 4.3–6.3 × in fig. 13: the network should ship the KV that each head can actually consume, not the dense tensor that a legacy interface expects. 

A serving engine designed around per-head attention would expose this granularity end-to-end . Its KV manager would maintain per-(layer , head ) page tables, allowing local heads to allocate only sink/recent pages and global heads to allocate full-context pages. Its attention kernel API would be varlen by default, taking per-head sequence lengths directly rather than constructing a dense rectangle and masking en-tries away. Its scheduler would account for heterogeneous head costs: global heads are bandwidth-heavy and should remain in fast HBM, while local heads can be cached, evicted, transferred, or tiered with much lower cost. Under GQA, the scheduler should also treat a KV head and its query-head fanout as an explicit scheduling object, rather than relying on 15 implicit broadcast inside a monolithic kernel. SegPagedAt-tention is one concrete instance of this engine direction. It does not change the sparse-attention semantics; it changes the physical contract between the cache manager and the kernel. Each head owns a compact page list, and the fused-varlen ker-nel consumes those ragged lengths without an additive mask. This is what allows the same head-class plan to become real compute, bandwidth, and capacity savings: 2.5–21.0 × faster decode over dense+mask, 6.3–23.3 × faster prefill, and 4.7– 7.8 × more concurrent sessions per GPU. The key point is that these gains are not independent optimizations; they are all consequences of making the head dimension visible to the storage layer, the kernel layer, and the scheduler. This perspective also explains the remaining gap be-tween RedKnot’s algorithmic savings and end-to-end sys-tem speedup. In the current prototype, parts of the serving stack still assume dense layer-level KV objects. As a result, some savings are realized as FLOP or byte reductions, while others are diluted by packing, transfer, scheduling, or dense-layout compatibility costs. A clean per-head engine would remove these impedance mismatches: cached chunks would be stored in their native per-head form, PD transfer would operate on head-class payloads, and attention would be dis-patched through fused varlen kernels without re-expansion. We view this as the natural next step for long-context RAG serving: once attention sparsity is per-head, the serving en-gine must be per-head as well. 

6.2 Position-independent Cache as the Default Cache Contract 

The second engine-level limitation is the cache contract itself. Modern LLM serving systems primarily expose a prefix cache: if two requests share the same leading token sequence, the engine reuses the KV for that prefix and prefills only the divergent suffix. This abstraction matches chat histories, but it is too narrow for RAG and agent serving. The same document chunk, tool output, or code snippet often reappears after a different user query or in a different retrieval order, so a prefix cache treats reusable content as a miss. 

Position-independent KV reuse changes the cache key from “the prefix of this request” to “the reusable content chunk”. The cache object, however, cannot be a single dense KV tensor with a binary hit/miss bit: a chunk’s KV is not equally valid for every head after it is moved behind a new pre-fix. Some heads are prefix-sensitive and must be recovered; others remain local and can be reused as-is. RedKnot’s head-class plan therefore attaches per-head validity and restore metadata to cached chunks, and SegPagedAttention stores the resulting KV object in the layout the kernel can consume directly. Making this contract native would also simplify sys-tem policy. Cache admission would prioritize high-reuse chunks rather than long shared prefixes; eviction would ac-count for per-head footprint and recovery cost; distributed KV stores would route content-addressed chunks across pre-fill workers; and PD disaggregation would transfer only the head-class payload needed by the decode side. We there-fore view position-independent, head-aware KV reuse as a cache-system primitive, not merely an inference optimization layered on top of prefix caching. 

7 Conclusion 

RedKnot revisits position-independent KV cache reuse by aligning its recovery, compute, and storage granularities with the per-head sparsity structure of the workload. It recovers cached KV at the granularity of attention heads rather than tokens, materializes per-head sparsity through SegPagedAt-tention so that every head stays on the FlashAttention fast path, and applies token-level Sparse FFN to attack the short-context FFN bottleneck that no attention-side technique can reach. Across three models, six QA datasets, and context lengths from 8 K to 128 K, RedKnot delivers up to 3.54× TTFT speedup and 4.7–7.8× more concurrent sessions per GPU while cutting prefill FLOPs by 79.5%, with end-to-end accuracy matching or exceeding the dense baseline.
