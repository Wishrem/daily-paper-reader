## 摘要
LLM服务日益受到来自智能体、推理模型和扩展对话的长且动态的解码工作负载主导。当突发性长上下文需求超过部署容量时，现有服务系统通常通过启动带有模型副本的额外服务实例进行水平扩展。这种实例级弹性仅通过提供另一份完整的模型副本增加KV容量，继承了启动延迟、内存开销和批次碎片化问题。我们提出KernelFlume，一种以解码为中心的架构，将稳定的投影/FFN路径与核心注意力计算解耦：权重节点执行密集的投影/FFN内核，而无权重注意力节点存储令牌范围KV分区，并根据请求状态需求扩展。为实现这种分离的弹性，KernelFlume维护一个路由表，将令牌范围映射到注意力节点端点。它在令牌边界处更新路由，并使用主机可见的图信号在捕获的CUDA图之外驱动预先注册的UCX端点通信。为保持解耦后低每令牌延迟，KernelFlume将查询优先的核心注意力调度与层间内核流水线结合，用本地投影/FFN工作重叠远程注意力和通信。在真实GPU测试平台（节点内A6000和跨节点H100）上，针对服务Llama-3.1-8B的动态长上下文智能体工作负载，KernelFlume在A6000上维持约74毫秒、在H100上约34毫秒的平坦p99 TPOT（每次输出令牌时间），同时相对于使用ServerlessLLM（一种最先进的实例启动方法）的全实例弹性扩展，每百万输出令牌成本分别降低高达32%和61%。在仿真中，以更大模型规模重放相同轨迹，相较于ServerlessLLM成本降低56–66%，使用更便宜的异构注意力节点硬件时扩大到80–85%，并持续到百万令牌上下文范围。

## Abstract
LLM serving is increasingly dominated by long and dynamic decode workloads from agents, reasoning models, and extended conversations. When bursty long-context demand exceeds deployed capacity, existing serving systems typically scale out by launching additional serving instances with model replicas. This instance-level elasticity increases KV capacity only by provisioning another full copy of the model, inheriting startup latency, memory overhead, and batch fragmentation.   We present KernelFlume, a decode-centric architecture that disaggregates the stable projection/FFN path from core-attention computation: weight nodes execute dense projection/FFN kernels, while weightless attention nodes store token-range KV partitions and scale with request-state demand. To make this separation elastic, KernelFlume maintains a routing table that maps token ranges to attention-node endpoints. It updates routes at token boundaries and uses host-visible graph signals to drive pre-registered UCX endpoint communication outside the captured CUDA Graph. To preserve low per-token latency after disaggregation, KernelFlume combines query-first core-attention dispatch with inter-layer kernel pipelining, overlapping remote attention and communication with local projection/FFN work. On real GPU testbeds (intra-node A6000 and cross-node H100), under a dynamic long-context agentic workload serving Llama-3.1-8B, KernelFlume sustains flat p99 TPOTs of ~74 ms on A6000 and ~34 ms on H100, while lowering cost per million output tokens by up to 32% and 61%, respectively, relative to full-instance elastic scaling with ServerlessLLM, a state-of-the-art instance-startup method. Replaying the same trace at larger model scale in simulation projects a 56--66% cost reduction over ServerlessLLM, widening to 80--85% with cheaper heterogeneous attention-node hardware and persisting into the million-token context range.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：LLM（大语言模型）在推理时，解码阶段因 KV 缓存随序列增长而快速膨胀，导致 GPU 显存不足。现有弹性扩展方法（如 ServerlessLLM）通过复制完整模型实例（包括权重）来增加 KV 容量，带来三大缺陷：
  - 实例启动延迟高（无法适应解码 SLO）；
  - 每个新增实例都携带完整权重副本，浪费 HBM；
  - 批次分裂导致 GEMM 利用率下降。
- **整体含义**：需要一种**更细粒度的弹性单元**——只扩展 KV 缓存，而不复制模型权重。KernelFlume 通过“解耦投影/FFN 与注意力”，将权重固定在大批量 GPU 上，而将 KV 缓存放置在可独立扩展的“无权重注意力节点”上，以轻量级路由更新替代全实例启动。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- **解耦架构**：将解码阶段拆分为两个节点类型：
  - **权重节点（W 节点）**：负责 QKV 投影、O 投影、FFN，保留模型权重；
  - **无权重注意力节点（A 节点）**：仅存储 KV 分区并计算局部注意力，不携带任何权重。
- **弹性单元**：添加/删除 A 节点，仅改变路由表映射，不改动 W 节点的 CUDA Graph，实现轻量级在线弹性。

### 关键技术细节
1. **弹性通信与路由表**：
   - 用**路由表**将 token 范围映射到 UCX 端点，代替固定 NCCL 集体通信。
   - W 节点执行 CUDA Graph，在 GPU 侧写入 host-visible 信号；CPU 轮询线程读取信号后，根据当前路由表发起 UCX 点对点发送。
   - 端点预注册，路由更新仅需 7.2 μs（实测）。
2. **查询优先注意力调度（QFA）**：
   - 在 QKV 投影中，一旦 Q 就绪就立即发送到持有历史 KV 的 A 节点，让其立即开始注意力计算，与 W 节点剩余的 K/V 投影重叠。
   - 尾 A 节点则等待 (K_new, V_new) 后追加并计算。
3. **层间内核流水线**：
   - 将解码批处理拆分为微批次（microbatch），使 W 节点的投影/FFN 与 A 节点的注意力在相邻微批次间重叠。
   - 流水线吞吐量接近 2× 序列执行上限（实测 1.87–1.94×）。
4. **弹性扩展策略**（算法 1）：
   - 每个 A 节点的 KV 字节预算由显存容量和 TPOT SLO 共同决定：\( C_{\text{thresh}} = \min(\beta_{\text{mem}}C_{\text{kv_cap}}, \beta_{\text{slo}} T_{\text{max}}^A \cdot BW_{\text{HBM}}) \)
   - 预测尾节点填充时间 \( \tau_{\text{fill}}^{(i)} \)，当剩余时间小于预启动时间时，后台预启动新 A 节点；达到水印后以路由切换激活。

## 6. 主要结论与发现

1. **弹性效果**：KernelFlume 在真实动态负载下保持平坦的 p99 TPOT（~74 ms A6000, ~34 ms H100），SLO 达标率 100%。
2. **成本降低**：相比 ServerlessLLM，每百万输出 token 成本在 A6000 上降低 21–32%，在 H100 上降低 27–61%。
3. **大模型仿真**：Llama-70B 仿真成本降低 56–66%，配合异构硬件（H20 作为 A 节点）扩大至 80–85%。
4. **超长上下文**：405B 模型在 1M–100M token 范围内，成本降低 65–72%。
5. **消融收益**：QFA 使解耦开销几乎消失（GQA 恢复 2–3%，MHA 恢复 5–7%）；层间流水线使吞吐量逼近 2× 上限。

## 7. 优点

- **弹性粒度细**：路由更新仅 7.2 μs，远低于实例启动（ms 级），不打断飞行中解码。
- **内存效率高**：新增 A 节点为零权重，全部 HBM 用于 KV，显存利用率远高于全实例扩展。
- **解耦开销低**：通过 QFA 和流水线，解耦后 TPOT 与无解耦基线基本持平，甚至更快（MHA 模型）。
- **支持异构硬件**：A 节点可使用更便宜、带宽型 GPU（如 H20），显著降低总成本。
- **设计简洁**：在现有框架（vLLM）上扩展，CUDA Graph 无需重捕获，与预填充优化兼容。

## 8. 不足与局限

- **实验对比不全面**：未与 LoongServe 等动态 CP 方案进行定量对比，仅定性讨论。
- **收缩机制未详细评估**：论文主要关注扩展（scale-up），对于 A 节点释放（scale-down）策略和开销分析较简单（仅给出 8.3 μs 路由移除）。
- **可靠性考虑不足**：未讨论 A 节点故障恢复、KV 复制、迁移压缩等生产环境需求。
- **极端长上下文验证仅仿真**：1M+ token 场景未在真实硬件上测试，仿真假设可能引入偏差。
- **CPU 轮询可能成为瓶颈**：虽然论文称轮询延迟仅数微秒，但在极大规模部署中可能受 CPU 频率影响，未深入评估。
- **通用性局限**：仅针对解码阶段解耦，不改变预填充/前缀缓存策略；路由策略当前仅基于序列范围分区，未考虑其他维度。
