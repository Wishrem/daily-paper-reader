## 摘要
我们提出了 Nemotron-Labs-3-Puzzle-75B-A9B，这是 Nemotron-3-Super 的一个压缩变体，专为交互式部署优化。我们设计该模型以在高用户吞吐量约束下最大化服务器吞吐量。在单个 8xB200 节点上的交互式服务负载中，Puzzle-75B-A9B 在匹配的用户吞吐量约束下，实现了比 Nemotron-3-Super 约高 2 倍的服务器吞吐量。在单个 H100 GPU 上的超长上下文部署中，压缩模型将 100 万 token 的并发度从 1 个请求提升至 8 个请求。Puzzle-75B-A9B 通过多阶段流水线构建，该流水线结合了迭代 Puzzle 压缩框架与知识蒸馏、强化学习、量化和多 token 预测头。压缩过程联合优化了异构 MoE 剪枝、活跃参数预算和 Mamba 剪枝，以提高推理效率同时保持模型质量。我们在广泛的推理、编码、多语言、长上下文和智能体基准上评估了 Puzzle-75B-A9B。尽管大幅压缩，该模型在多种任务上相对于父模型仍保持了很强的下游准确性。这些结果表明，大型混合 MoE 模型可以在保持强大下游能力的同时，大幅优化部署效率。

## Abstract
We present Nemotron-Labs-3-Puzzle-75B-A9B, a compressed variant of Nemotron-3-Super optimized for interactive deployment. We designed the model to maximize server throughput under high user throughput constraints. In interactive serving workloads on a single 8xB200 node, Puzzle-75B-A9B achieves approximately 2x higher server throughput than Nemotron-3-Super at matched user throughput constraints. In ultra-long-context deployment on a single H100 GPU, the compressed model increases 1M-token concurrency from 1 request to 8 requests. Puzzle-75B-A9B is constructed using a multi-stage pipeline that combines the Iterative Puzzle compression framework with knowledge distillation, reinforcement learning, quantization, and a Multi-Token Prediction head. The compression process jointly optimizes heterogeneous MoE pruning, active parameter budget, and Mamba pruning to improve inference efficiency while preserving model quality. We evaluate Puzzle-75B-A9B on a broad suite of reasoning, coding, multilingual, long-context, and agentic benchmarks. Despite substantial compression, the model retains strong downstream accuracy relative to the parent model across a wide range of tasks. These results demonstrate that large hybrid MoE models can be substantially optimized for deployment efficiency while maintaining strong downstream capability.

## 2. 提出的方法论

### 2.1 核心思想
采用**多阶段压缩-恢复流水线**，包括：
- **迭代Puzzle压缩**：将单步Puzzle框架扩展为交替进行渐进式结构压缩和短期知识蒸馏（KD）的迭代过程，适应压缩过程中模型内部表示的变化，更精准地分配容量。
- **知识蒸馏（KD）**：在压缩阶段和恢复阶段均使用原模型作为教师，指导学生模型匹配logits，短/长上下文分别训练。
- **强化学习（RL）恢复**：针对编码等压缩敏感任务进行RL微调（两阶段SWE-RL），并对多个学习率运行的权重进行平均。
- **量化**：分别生成FP8（Hopper）和NVFP4（Blackwell）量化检查点，优化算子精度分配（表2）。
- **多token预测（MTP）**：利用共享MTP头实现推测解码，并通过针对性训练缓解训练-推理分布不匹配，提升接受长度。

### 2.2 关键技术细节

- **迭代Puzzle（Iterative Puzzle）**：
  - 算法：从原始模型M0开始，每轮执行：在当前模型Mr−1上构建搜索空间 → 利用Puzzle进行增量剪枝（混合整数规划求解）→ 教师KD愈合 → 得到Mr。重复直到满足部署目标。
  - 剪枝维度：
    - **MoE中间通道剪枝**：对每个路由专家按输出贡献剪至统一大小（层间可异构）。
    - **Top-k减少**：每层活跃专家数从22降至4-18（层间异构）。
    - **Mamba SSM状态剪枝**：根据贡献度评分将SSM状态维度从128剪至96，加速解码阶段约1.2-1.3倍。
  - **剪枝策略选择**：MoE采用贡献度引导的通道选择，Mamba贡献度平均绝对激活与后续权重范数乘积（独立贡献），与随机基线对比（图7）。

- **恢复阶段**：
  - **短上下文KD**：32K序列，24B+43.2B+52.8B tokens（三轮对应不同压缩目标）。
  - **长上下文KD**：128K→512K，100B tokens，全局batch 16M tokens。
  - **RL**：SWE-RL分两阶段（单步工具比较 + 多轮沙箱），KL penalty=0，学习率扫掠后权重平均。

- **MTP改进**：共享MTP头继续训练，使内部草案模型对自回归使用更鲁棒，接受长度提升25%-30%。

1. **Puzzle-75B-A9B在精度上接近Super（综合平均约98%），但在交互式服务中总吞吐提升约2倍，在1M-token单GPU并发从1增加至8**。
