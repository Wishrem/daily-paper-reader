## 摘要
新兴的LLM服务越来越多地托管多个稀疏MoE模型，然而大多数模型收到的请求稀疏且处于冷状态。这带来了GPU内存问题：模型权重是稳定的且由模型决定，而KV缓存是瞬态的且由需求决定。由于冷模型很少同时达到峰值KV缓存需求，为每个模型预留最坏情况的KV容量会浪费内存；共享KV缓存池可以按需配置聚合的活跃需求。然而，当权重和KV缓存仍处于单一GPU内存池中时，KV缓存共享是不够的。静态权重与动态KV缓存竞争，在冷、低并发流量下，KV头限制的注意力仅暴露复用的KV容量的一小部分，导致GPU内存利用率低和长上下文支持弱。我们提出了CrossPool，一种用于冷MoE模型的服务引擎，它将FFN权重和KV缓存分离到两个GPU内存池中：一个权重池，合并冷模型间的FFN权重；一个KV缓存池，动态服务活跃请求，同时使注意力局部于KV缓存。CrossPool结合了KV缓存规划器和虚拟化器、隐藏隐藏状态传输的逐层流水线调度器，以及通过控制降级减少CPU-GPU控制开销的持久内核。通过高效的GPU内存池化，CrossPool支持突发的长上下文请求，并优于基于kvcached的最先进的多LLM服务系统，将P99 TBT最多降低10.4倍。

## Abstract
Emerging LLM services increasingly host many sparse MoE models, yet most models receive sparse requests and remain cold. This creates a GPU memory problem: model weights are stable and model-determined, while KV-cache is transient and demand-determined. Because cold models rarely reach peak KV-cache demand at the same time, reserving worst-case KV capacity per model wastes memory; a shared KV-cache pool can instead provision aggregate active demand. However, KV-cache sharing is not sufficient when weights and KV-cache remain in a monolithic GPU memory pool. Static weights compete with dynamic KV-cache, and KV-head-limited attention under cold, low-concurrency traffic exposes only a fraction of replicated KV capacity, leading to low GPU memory utilization and weak long-context support. We present CrossPool, a serving engine for cold MoE models that separates FFN weights and KV-cache into two GPU memory pools: a weights pool that consolidates FFN weights across cold models, and a KV-cache pool that dynamically serves active requests while keeping attention local to KV-cache. CrossPool combines a KV-cache planner and virtualizer, a layer-wise pipeline scheduler that hides hidden-state transfers, and persistent kernels with control lowering to reduce CPU-GPU control overhead. With efficient GPU memory pooling, CrossPool underpins bursty long-context requests and outperforms the state-of-the-art kvcached-based multi-LLM serving system, reducing P99 TBT by up to 10.4x.

# CrossPool: 基于KV缓存与权重分离的冷门MoE模型高效多LLM服务

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究背景**：大型语言模型（LLM）服务中，大多数模型是“冷模型”——请求稀疏、并发度低，但需要长期驻留GPU内存。模型权重是稳定的、由模型决定，而KV-cache是瞬态的、由需求决定。在单一的GPU内存池中，静态的权重与动态的KV-cache竞争内存，导致GPU内存利用率低下。
- **核心问题**：现有系统（如MuxServe、kvcached）虽然实现了多模型共享GPU内存，但仍将权重与KV-cache放在同一内存池中，导致：
  - 为每个模型预留最坏情况的KV-cache容量造成浪费；
  - 在KV头有限的注意力机制（如MQA、MLA）下，低并发时无法充分利用复制的KV-cache容量，长上下文支持弱。
- **整体目标**：通过将FFN权重与KV-cache分离到不同GPU内存池，在保证推理性能的前提下，显著提升GPU内存利用率，支持突发长上下文请求。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将FFN权重（占MoE模型绝大部分参数）合并到一个**权重池**，将KV-cache（与其他非FFN模块）放在另一个**KV-cache池**。注意力计算局部于KV-cache池，FFN计算在权重池执行。池间仅传输隐藏状态（大小与batch tokens和隐藏维度相关），而非KV-cache张量，从而避免传输与上下文长度成比例的数据。
- **关键技术细节**：
  - **KV-cache规划器与虚拟化器（KV-cache Planner and Virtualizer）**：
    - 离线规划器：基于每个模型的访问模式（到达率、prompt/输出长度、服务时间等），以及每个token所需的KV字节数，使用蒙特卡洛方法估算全局KV-cache需求的高分位数（如P95/P99），确定共享KV-cache池的页面预算。
    - 在线虚拟化器：利用CUDA VMM API为每个模型保留虚拟KV地址范围，按需映射物理页面。注意力操作通过虚拟化分页KV-cache接口访问，物理页分配在慢路径上进行。
  - **逐层流水线调度器（Layer-wise Pipeline Scheduler）**：
    - 维护两个in-flight批次，分别在不同模型的不同层上交替执行注意力和FFN。当一批次在KV-cache池执行注意力时，另一批次的隐藏状态已在权重池执行FFN，从而隐藏池间传输延迟。
    - 支持批次提前完成，无需全局层同步，适应异构冷模型。
  - **持久内核与控制降级（Persistent Kernels and Control Lowering）**：
    - 分别捕获注意力和FFN的计算图，在预热时编译。
    - 启动GPU驻留的持久内核，不断轮询请求队列，调度计算图、触发池间通信，避免CPU干预层间调度，大幅减少CPU-GPU控制开销。
- **公式/算法流程（文字说明）**：
  - 规划器：对每个模型M，请求i到达时间Ai，prompt tokens Op,i，output tokens Od,i，解码驻留时间Ti。在时间t，活跃KV tokens Qi(t) = (Op,i + Od,i * (t-Ai)/Ti) * indicator{0≤t-Ai<Ti}。模型M的KV使用KM(t) = Σ κ(M) Qi(t-Ai)。聚合池需求Kpool(t) = ΣM KM(t)。规划器通过蒙特卡洛仿真得到Kpool的P95/P99值，除以页面大小得到页面预算。
  - 在线服务：调度新请求到具有最大空闲KV空间的可用DP rank；虚拟化器管理物理页分配。
