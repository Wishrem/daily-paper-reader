## 摘要
现代CPU越来越多地集成矩阵扩展，例如Arm可扩展矩阵扩展（SME），可在CPU内提供高吞吐量矩阵执行。然而，对于大语言模型推理，这些单元并不能普遍取代传统CPU核心：预填充、解码、注意力机制和KV缓存操作表现出不同的算术强度、向量行为和布局要求，而SME单元和CPU核心仍共享内存带宽。本文通过基于屋顶线模型的SME CPU特性化研究这种不匹配，并利用所得模型指导算子级执行选择。我们提出SMEPilot，一个为大语言模型推理引擎，针对每个算子形状选择纯CPU、纯SME或SME+CPU协同执行。SMEPilot将矩阵工作以瓦片粒度划分给SME和CPU核心，在注意力中重叠SME适合的矩阵阶段与CPU适合的向量阶段，并维护布局状态以便压缩张量表示得以重用而非在关键路径上反复重建。在手机、PC和服务器平台上，针对Llama-3.2-3B、Qwen3-4B和Qwen3-30BA3B，SMEPilot将端到端推理性能提升高达3.94倍。

## Abstract
Modern CPUs increasingly integrate matrix extensions, such as Arm Scalable Matrix Extension (SME), that provide high-throughput matrix execution within the CPU. For LLM inference, however, these units are not a universal replacement for conventional CPU cores: prefill, decode, attention, and KV-cache operations expose different arithmetic intensities, vector behavior, and layout requirements, while SME units and CPU cores still compete for shared memory bandwidth. This paper studies this mismatch through a roofline-based characterization of SME-enabled CPUs and uses the resulting model to guide operator-level execution choices. We present SMEPilot, an LLM inference engine that selects CPU-only, SME-only, or cooperative SME+CPU execution for each operator shape. SMEPilot partitions matrix work across SME and CPU cores at tile granularity, overlaps SME-suitable matrix stages with CPU-suitable vector stages in attention, and maintains layout state so packed tensor representations are reused rather than repeatedly rebuilt on critical paths. Across Llama-3.2-3B, Qwen3-4B, and Qwen3-30BA3B on phone, PC, and server platforms, SMEPilot improves end-to-end inference performance by up to 3.94$\times$.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
基于多天花板Roofline模型，对每个算子计算其算术强度，将其归类为：
- **内存受限区域**（CPU-only）
- **后端非对称脊点穿越区域**（SME-only）
- **计算受限区域**（混合SME+CPU执行）

然后根据分类生成每个算子的执行计划，包括后端选择、线程分配和瓦片划分比例。

### 关键技术细节

1. **统一Roofline模型**：
   - 共享内存带宽屋顶（B），多个计算天花板（纯CPU、纯SME、混合SME+CPU）。
   - 对于算子o，计算其算术强度 I_o = 浮点操作数 / 有用内存流量。
   - 性能估计：R_h(o) = min(P_h, B * I_o)。不同目标有不同脊点 I_ridge_h = P_h/B。
   - 根据I_o与脊点的关系确定执行模式。

2. **瓦片级工作划分**（Tile-level Work Partitioning）：
   - 对于计算受限的矩阵运算（如大prefill GEMM），将输出维度划分为瓦片，分配给SME和CPU核心同时计算。
   - 选择较大的输出维度进行划分，以避免同步开销，保持瓦片对齐。
   - MoE专家层面：根据每个专家接收的token数量决定是否启用SME（token少用CPU，token多用SME或混合）。

3. **阶段感知的流水线执行**（Phase-aware Pipeline Execution）：
   - 针对注意力机制：将SME友好的矩阵阶段（QK^T, PV）与CPU友好的向量阶段（softmax、mask）进行流水线化。
   - 每个注意力瓦片依次经过QK^T (SME) → softmax (CPU) → PV (SME)，不同瓦片可同时在不同阶段执行，消除气泡时间。
   - 对掩码（causal/sliding window）进行感知，跳过完全掩码的瓦片以减少无效计算。

4. **布局感知运行时**（Layout-aware Runtime）：
   - 跟踪张量布局状态，在模型加载时一次性地将静态权重打包为SME友好布局，避免每次推理重复转换。
   - 对于在线生成的激活（如KV cache），在生产者侧直接写入SME友好布局（如分组打包），使得下游SME注意力无需重新打包。
   - 消除布局转换对关键路径的影响。

5. **执行计划生成**：
   - 离线分析器：测量目标CPU的SME矩阵吞吐量、CPU吞吐量、内存带宽等硬件指标（一次测量，多次复用）。
   - 在线规划器：根据当前请求的形状（token数、batch size等）和模型结构，使用Roofline模型为每个算子选择后端、分配SME/CPU线程数、计算最佳划分比例（对齐到内核瓦片大小）。
