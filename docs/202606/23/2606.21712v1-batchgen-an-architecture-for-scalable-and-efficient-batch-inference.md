## 摘要
批量推理已成为AI计算的核心模式，然而现有的推理引擎仍然依赖于为交互式服务设计的执行模型。当扩展到数百万个序列时，批量工作负载暴露了两个基本需求：处理仅在运行时出现的极端序列间和序列内负载变化的能力，以及在大型GPU集群中维持高利用率的能力。现有系统无法满足这些需求，损失了相当大比例的可达吞吐量。

我们为批量推理引入了一种新的架构基础：序列协程计算模型，它将每个序列表示为一个细粒度的、事件驱动的协程。该模型暴露了富有表现力的原语，允许运行时动态重组工作，从而实现更大的专家级批量、缓解掉队者、跨设备重新分配工作，甚至在成本效益高或内存受限的GPU上也能维持高利用率。基于这一抽象，我们实现了BatchGen，一个在集群规模上使用协程模型的、可投入生产的系统。在128-GPU集群上，BatchGen将批量完成时间减少了高达2.3倍，在内存受限的加速器上，其性能比最强的卸载基线高出高达9.6倍。我们将在https://github.com/batchgen-project/batchgen开源BatchGen。

## Abstract
Batch inference has become a central mode of AI computation, yet existing inference engines still rely on execution models designed for interactive serving. When scaled to millions of sequences, batch workloads reveal two fundamental requirements: the ability to handle extreme inter- and intra-sequence load variation that emerges only at runtime, and the ability to sustain high utilization across large fleets of GPUs. Existing systems fail to meet these requirements, losing substantial fractions of achievable throughput.   We introduce a new architectural foundation for batch inference: the sequence coroutine compute model, which represents each sequence as a fine-grained, event-driven coroutine. This model exposes expressive primitives that allow the runtime to reorganize work dynamically, enabling larger expert-level batches, mitigating stragglers, reallocating work across devices, and maintaining utilization even on cost-effective or memory-constrained GPUs. Building on this abstraction, we implement BatchGen, a production-ready system that uses the coroutine model at cluster scale. On a 128-GPU cluster, BatchGen reduces batch completion time by up to $2.3\times$, and on memory-constrained accelerators it outperforms the strongest offloading baseline by up to $9.6\times$. We will open-source BatchGen at https://github.com/batchgen-project/batchgen

# BatchGen 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

- **研究背景**：批量推理（batch inference）已成为 AI 计算的核心模式，广泛应用于离线推理、合成数据生成、模型评估、测试时扩展（test-time scaling）和强化学习 rollout 等场景。不同于交互式推理的低延迟优化，批量推理优化的是**批量完成时间（Batch Completion Time, BCT）**，操作于大规模序列（百万级）。
- **核心问题**：现有推理引擎（如 vLLM、SGLang、TensorRT-LLM）继承自交互式服务的**延迟优先执行模型**，静态绑定序列到特定 GPU，无法处理批量推理中两个关键挑战：
  - **序列内负载变化**：混合专家（MoE）模型中，每个 token 仅激活少数专家，导致每专家的批量尺寸远低于 GPU 饱和吞吐量，GPU 利用率往往仅 50% 左右。
  - **序列间负载变化**：测试时扩展和深度推理产生长尾序列（如 DeepSeek-R1 中 P99 输出长度是 P95 的 3.78 倍），这些拖后腿的序列主导 BCT，使大多数 GPU 空闲。
- **整体含义**：急需一种**批量原生**的推理引擎，能动态适应运行时负载变化，维持大规模 GPU 集群的高利用率。

## 2. 方法论：核心思想、关键技术细节

### 核心思想：事件驱动的序列协程计算模型

将每个序列建模为**细粒度、事件驱动的协程**，可在模块边界暂停、组合、分割、迁移，从而动态重组计算。

### 关键技术细节

1. **序列协程抽象**：每个序列携带完整状态（KV cache、隐藏状态等），提供四个原语：
   - `yield`：挂起协程，释放 GPU，检查点状态。
   - `combine`：合并多个挂起序列形成一个批次，用于增大稀疏模块（如 MoE）的批大小。
   - `partition`：将单个长序列通过张量并行分布到多个 GPU，加速完成。
   - `migrate`：异步转移序列状态到其他 GPU，实现负载均衡。

2. **调度算法**（以 MoE 模型为例）：
   - **静态 yield 点选择**：在每个 attention 层后设置 yield 点，允许在 MoE 层前 combine 更多序列形成大 batch；跨前向传播的 yield 点根据运行时内存压力动态决定。
   - **前向传播调度（Algorithm 1）**：在 attention 层将 batch 拆分为子批次（大小为 B_attn），每个子批次执行 attention 后 yield 并异步将 KV cache 卸载到主机；然后在 MoE 层将所有子批次的隐藏状态 combine 成一个大 batch（大小为 B_moe）执行。
   - **全局调度循环（Algorithm 2）**：节点不断从主机内存中选择 inactive 序列形成活跃 batch，执行前向传播与动态序列管理；当序列完成或内存不足时，yield 并 refill；当剩余少数长尾序列时，触发 partition 将计算并行化到空闲 GPU。

3. **内存模型**：
   - 主机内存作为统一存储，持有所有序列的 KV cache 和模型参数。
   - GPU 内存分为驻留参数（小，如 LayerNorm）和缓冲区（参数预取/KV 暂存）。
   - 解码阶段采用**分页 KV cache**，每序列仅预留两页，按需分配，最大化并发。
   - 预填阶段（prefill）通过异步卸载每层 KV cache 到主机，GPU 内存仅保留两层 KV cache，降低峰值需求。

4. **调度计划自动优化**：通过轻量级 profiling 单层性能，构建执行 DAG，搜索最小化关键路径的配置（batch 大小、缓冲区分配、yield 点位置）。

## 6. 主要结论与发现

1. **性能提升**：
   - 128 GPU 集群上，BatchGen 降低 BCT 最高 **2.3×**（解码重型负载）。
   - 内存受限 GPU（24 GB）上，比最强卸载基线 **快 9.6×**。
   - 相比 SGLang-Optimized，离线推理速度提升 1.25–1.85×，测试时扩展处理更多序列 1.25–1.75×。
   - 只能运行 Kimi-K2（1T 参数）于 8×H20 上，其他系统 OOM。

2. **批量支持能力**：BatchGen 可支持比基线大得多的序列同时解码（H20 上 1800+ vs 8-16 序列），通过 offloading 实现。

3. **长尾问题缓解**：通过 partition 将尾部拖后腿序列分布到空闲 GPU，在 RL rollout 中减少 5–10% 每次迭代时间。

4. **自动调度优于手动调优**：PD 分离需 exhaustive 调优比例（性能变化 3.6×），BatchGen 无需调优即可超越最佳 PD 配置。

## 7. 优点（方法与实验设计亮点）

- **创新性**：首次将事件驱动协程抽象引入批量推理，类比从 Apache 到 NGINX 的范式转变。
- **高效原语设计**：yield/combine/partition/migrate 开销极低（<5 μs ~ 秒级，但仅在尾部触发），支持隐藏传输延迟。
- **统一处理预填和解码**：同一框架内联合调度两个阶段，避免独立优化。
- **自动配置优化**：通过性能建模与 DAG 搜索自动选择合适的 batch 大小和缓冲区，免去手动调整。
- **实验覆盖面广**：从小型 GPU（单卡 24 GB）到大规模集群（128 卡），从离线到 SLO 约束在线，考虑多种负载特性。
- **消融与对比严谨**：通过 BatchGen*（使用基线内核）证明调度收益独立于内核优化。
- **生产就绪**：提供 OpenAI 兼容 API、cold-start 优化、故障恢复，并已在生产集群部署。
