## 摘要
新兴的LLM服务越来越多地托管多个稀疏的MoE模型，然而大多数模型接收到的请求稀疏且处于冷状态。这造成了GPU内存问题：模型权重是稳定的且由模型决定，而KV缓存是瞬时的且由需求决定。由于冷模型很少同时达到KV缓存需求的峰值，为每个模型预留最坏情况下的KV容量会浪费内存；一个共享的KV缓存池可以按需配置聚合的活跃需求。然而，当权重和KV缓存仍处于单一GPU内存池中时，KV缓存共享是不够的。静态权重与动态KV缓存竞争，并且在冷、低并发流量下，受限于KV头的注意力仅暴露了复制KV容量的一部分，导致GPU内存利用率低和长上下文支持弱。我们提出了CrossPool，一种针对冷MoE模型的服务引擎，它将FFN权重和KV缓存分离到两个GPU内存池中：一个权重池，用于整合冷模型的FFN权重；一个KV缓存池，动态服务活跃请求，同时将注意力保持在KV缓存本地。CrossPool结合了KV缓存规划器和虚拟化器、一个隐藏状态传输的逐层流水线调度器，以及通过控制降级减少CPU-GPU控制开销的持久化内核。通过高效的GPU内存池化，CrossPool支撑了突发性的长上下文请求，并超越了基于KV缓存的最先进的多LLM服务系统，将P99 TBT降低了高达10.4倍。

## Abstract
Emerging LLM services increasingly host many sparse MoE models, yet most models receive sparse requests and remain cold. This creates a GPU memory problem: model weights are stable and model-determined, while KV-cache is transient and demand-determined. Because cold models rarely reach peak KV-cache demand at the same time, reserving worst-case KV capacity per model wastes memory; a shared KV-cache pool can instead provision aggregate active demand. However, KV-cache sharing is not sufficient when weights and KV-cache remain in a monolithic GPU memory pool. Static weights compete with dynamic KV-cache, and KV-head-limited attention under cold, low-concurrency traffic exposes only a fraction of replicated KV capacity, leading to low GPU memory utilization and weak long-context support. We present CrossPool, a serving engine for cold MoE models that separates FFN weights and KV-cache into two GPU memory pools: a weights pool that consolidates FFN weights across cold models, and a KV-cache pool that dynamically serves active requests while keeping attention local to KV-cache. CrossPool combines a KV-cache planner and virtualizer, a layer-wise pipeline scheduler that hides hidden-state transfers, and persistent kernels with control lowering to reduce CPU-GPU control overhead. With efficient GPU memory pooling, CrossPool underpins bursty long-context requests and outperforms the state-of-the-art kvcached-based multi-LLM serving system, reducing P99 TBT by up to $10.4\times$.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：随着LLM服务部署越来越多的稀疏MoE模型，大多数模型请求稀疏，处于“冷”状态。这导致GPU内存使用矛盾：模型权重是稳定的、由模型决定；而KV-cache是瞬时的、由需求决定。冷模型的KV-cache峰值很少同时出现，为每个模型预留最坏情况下的KV容量会浪费内存；而共享KV-cache池可以按聚合活跃需求配置。但是，在单一GPU内存池中，静态权重与动态KV-cache竞争，且KV头数受限的注意力算法（如MQA、MLA）在低并发时只暴露了复制KV容量的一部分，导致GPU内存利用率低、长上下文支持弱。
- **核心问题**：现有系统（如MuxServe、kvcached）将权重和KV-cache置于同一GPU池，冷模型下KV-cache有效容量受限，无法高效支持突发长上下文请求。
- **整体含义**：需要将稳定权重与瞬态KV-cache解耦，通过专用池化提升冷MoE模型的内存利用率和长上下文服务能力。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将FFN权重和KV-cache分离为两个独立的GPU内存池：
  - **权重池**：整合多个冷模型的FFN权重（占MoE模型参数量主导）。
  - **KV-cache池**：动态服务活跃请求，并将注意力模块保持在本池中以实现本地KV访问；非FFN模块（注意力等）也留在KV-cache池。
  - 池间仅交换隐藏状态（大小受限于批次令牌数和隐藏维度，不随上下文长度增长），避免传输KV-cache张量。

- **关键技术细节**：
  1. **KV-cache规划器与虚拟化器**：
     - 离线规划器：基于每个模型的提示长度、输出长度、服务时间样本和到达率，通过蒙特卡洛分位数估计（P95/P99）计算共享KV-cache池的页面预算和并行计划。
     - 在线虚拟化器：利用CUDA VMM API预留虚拟KV地址范围，按需映射物理页面，注意力内核看到分页KV-cache接口。
  2. **逐层流水线调度器**：
     - 在KV-cache池和权重池之间按层粒度调度：一批次在KV-cache池执行注意力，另一批次的隐藏状态在权重池执行FFN，重叠计算和跨池传输。
     - 支持不同模型的不同层数和提前退出，无需全局层同步。
  3. **持久化内核与控制降级**：
     - 分别捕获注意力和FFN子图，利用CUDA图执行。
     - 持久化内核在GPU上持续运行，轮询GPU上的请求队列，调度图执行和跨池通信，避免CPU-GPU频繁控制切换。

- **公式/算法流程**（文字说明）：
  - 规划器：对于模型 $M$，请求 $i$ 的活跃KV增长函数 $Q_i(u)$，模型 $M$ 在时间 $t$ 的KV需求 $K_M(t) = \sum_i \kappa(M) Q_i(t-A_i)$，聚合池需求 $K_{\text{pool}}(t) = \sum_M K_M(t)$，以P95/P99分位数作为预算。
  - 解码执行流程：请求批次先执行注意力（KV-cache池）→ 隐藏状态传输到权重池 → 执行FFN → 结果传输回KV-cache池，进入下一层。通过流水线调度重叠不同批次的不同阶段。
