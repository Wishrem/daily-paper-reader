## 摘要
混合专家（MoE）架构将大型语言模型（LLM）扩展到数千亿参数。服务单个MoE模型需要多个GPU并行运行，通常通过张量并行（TP）或专家并行（EP）。最佳选择取决于运行中的请求数量：TP在低并发时更快，而EP在高并发时胜出。生产工作负载会不断跨越这一边界：在线服务会经历突发的高峰期，然后消退到平静期；强化学习推演开始时是高并发突发，然后衰减为长尾的落后任务。因此，当工作负载跨越到另一边时，固定任何一种布局都会损失性能。我们提出Moebius，一个服务系统，它在运行时在EP和TP之间切换，无需重启引擎或丢弃正在处理的请求。我们的关键洞察是，EP和TP是一个模型的两种布局，而不是两个模型：它们对字节相同的专家权重和KV缓存计算相同的函数，因此切换只改变每个秩拥有哪个切片。移动那些所有权改变的切片是唯一不可减小的开销，现代高带宽GPU互连使其足够快，可以在解码步骤之间完成，而无需排空正在处理的请求。Moebius保持每个并行度的运行时驻留，并在固定地址上使用融合的GPU到GPU传输内核重新分片专家权重和KV缓存的单一副本。在8个H200 GPU上服务Qwen3-235B-A22B时，Moebius在每个操作点都匹配更好的静态并行度，并在RL推演中跨步骤实现了1.16-1.25倍的提升。每次切换在215-434毫秒内完成，Moebius仅以2.4%的内存开销保持两种布局驻留。

## Abstract
Mixture-of-Experts (MoE) architectures scale large language models (LLMs) to hundreds of billions of parameters. Serving a single MoE model requires multiple GPUs operating in parallel, typically through tensor parallelism (TP) or expert parallelism (EP). The optimal choice depends on the number of in-flight requests: TP is faster at low concurrency, whereas EP wins at high concurrency. Production workloads cross this boundary continually: online serving sees bursty arrivals that subside into quiet periods, and reinforcement-learning rollouts begin as a high-concurrency burst that decays into a long tail of stragglers. Pinning either layout therefore forfeits performance when the workload crosses to the other side.   We present Moebius, a serving system that switches between EP and TP at runtime without restarting the engine or dropping in-flight requests. Our key insight is that EP and TP are two layouts of one model, not two models: they compute the same function over byte-identical expert weights and KV cache, so a switch changes only which rank owns each slice. Moving those owner-changed slices is the sole irreducible cost, and modern high-bandwidth GPU interconnects make it fast enough to do between decode steps without draining in-flight requests. Moebius preserves each parallelism's runtime resident, and reshards the single copy of expert weights and KV cache at fixed addresses with fused GPU-to-GPU transfer kernels. On 8x H200 GPUs serving Qwen3-235B-A22B, Moebius matches the better static parallelism at every operating point, and beats it on RL rollouts by 1.16-1.25x across steps. Each switch completes in 215-434 ms, and Moebius holds both layouts resident with only 2.4% memory overhead.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：混合专家（MoE）大语言模型（如Qwen3-235B、DeepSeek等）部署时需要多个GPU并行工作，通常采用张量并行（TP）或专家并行（EP）。
- **问题**：最优的并行策略取决于运行中的请求数量（并发度）。低并发时TP更快（低延迟），高并发时EP更优（高吞吐）。实际生产负载（在线服务突发的“高峰-低谷”模式、强化学习推演的“初始爆发→长尾衰变”模式）会动态跨越这一性能交叉点，固定使用TP或EP都会在另一方占优时损失性能。
- **整体含义**：需要一个能在运行时无缝切换TP与EP的MoE服务系统，以自适应负载变化，无需重启、丢请求或重建运行时。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **关键洞察**：EP和TP本质上是同一模型的不同布局，不是两个独立的模型。它们对**字节相同的专家权重和KV缓存**执行相同的函数，唯一的区别是**哪个GPU（rank）拥有哪个切片**。因此，并行度切换的不可削减开销仅仅是**移动所有权发生改变的数据切片**。
- 利用现代高带宽GPU互连（如NVLink）可以在解码步骤之间快速完成数据迁移，无需排空在途请求。

### 关键技术细节（三大机制）
1. **统一内存管理器（Unified Memory Manager, UMM）**
   - 启动时在每个GPU上分配一块大的连续缓冲区，用于存放专家权重、KV缓存页、请求缓冲和传输暂存空间。
   - 为每个并行度定义“张量视图”（alias），指向同一物理地址但形状/步幅不同。例如，对N层模型，TP将第i层映射到槽i，EP映射到槽i+1（多一个槽作为传输目标，避免覆盖）。
   - 所有地址固定不变，因此预捕获的CUDA图在切换后仍然有效。

2. **融合直接传输内核（Fused Direct Transfer Kernels）**
   - 针对权重和KV缓存分别实现专用内核，直接在GPU间通过NVLink将数据从源布局写入目标布局的固定槽位，无需中间暂存缓冲区或逐层的All-to-All。
   - 权重重分片：EP→TP先本地打包再All-to-All；TP→EP先All-to-All再本地重组。
   - KV缓存重分片：读取页表，将分散的块收集成连续块，通过All-to-All交换，再分散到目标页中。
   - 结果比NCCL集合通信快1.49倍（权重）和2倍以上（缓存），带宽利用率超过70%。

3. **运行时保留（Runtime Preservation）**
   - 同时保留两种并行度的CUDA图、通信组和注意力元数据。切换时仅交换指向活跃图的指针（亚毫秒级），而非重新构建。
   - 启动时通过一次空载切换捕获另一种模式的图，后续切换仅更新数据内容。

4. **切换策略（Switch Policy）**
   - 以全局在途请求数作为控制信号，由rank 0的协调器决定。
   - 非对称策略：TP→EP采用高/低阈值+滞后窗口+冷却时间，避免振荡；RL推演则使用相同阈值和窗口=1（单调减少）。
   - 切换前检查目标模式是否有足够KV容量（TP因头部复制容量较小），若不满足则取消并重试。

## 6. 论文的主要结论与发现

- **Moebius在每一个操作点都匹配更好的静态布局**：在静态负载扫描中，Moebius的延迟曲线始终取TP和EP的下包络线。
- **在线服务中，Moebius在爆发期接近EP的TTFT（峰值3.1s vs EP的2.0s），在平静期接近TP的TPOT（37ms vs EP的52ms）**，整体优于任一固定布局。
- **RL推演中，Moebius在每个步骤上都最快**，比后验最优静态布局快1.16-1.25倍（平均1.22倍），比最差静态布局快最多1.31倍。这是因为Moebius在爆发阶段用EP、长尾阶段用TP，匹配了两阶段的最优布局。
- **切换开销极低**：单次切换耗时215-434 ms，其中权重转移固定为152 ms（使用融合内核），KV缓存部分随占用率线性增长，但即使在满载下也在0.5秒内。
- **内存开销仅2.4%**：双模式缓冲区仅占每GPU 2.8 GB（通过缩小KV缓存容量获得），总内存占用与静态EP几乎相同（比TP低3.7 GB），因为大多数状态（专家权重、通信缓冲区）是共享的。
- **CUDA图保留避免了每秒到数秒的重建开销**：无图时解码延迟在低并发下慢6.95倍，保留两张图仅需36个图，内存开销可忽略。

## 8. 不足与局限

### 局限性
- **切换策略简单**：仅基于在途请求数，未考虑实时延迟或吞吐预测，可能在复杂负载下不够最优。

- **融合传输内核针对NVL8优化**：端口到其他互连（如InfiniBand）需额外工作。
- **内存开销虽小但非零**：2.4%的KV缓存减少可能影响极限并发容量。
