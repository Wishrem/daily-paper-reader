Abstract —Large Language Models (LLMs) have become in-tegral to modern applications, yet their deployment remains challenging. Beyond executing the models themselves, practical deployment must address cost efficiency, low latency, and optimal resource utilization. Conventional approaches typically assume that an entire model can be hosted on a single device, which does not hold in many real-world scenarios—particularly in Edge and Fog environments where device resources are constrained. In this paper, we introduce E2LLM, a framework designed to enable efficient LLM deployment in such resource-limited settings. Rather than simply partitioning a single model across all available devices, E2LLM replicates the full model across mul-tiple groups of devices (replicas) and applies model parallelism within each replica. Each replica is assigned a specialized role— PREFILL or DECODER—based on its efficiency in handling input and output tokens. This separation leverages the inherent differences between these two phases of LLM inference. To effectively organize devices, we utilize a Genetic Algorithm to form clusters that maximize system performance. Within each cluster, we apply Dynamic Programming to determine an optimal partitioning strategy that minimizes bottlenecks in model-parallel execution. Experimental results demonstrate that our approach adapts robustly to varying workloads, including scenarios with significant variation in input and output token lengths. Compared to the Splitwise baseline, E2LLM reduces average waiting time by over 50% under high-demand conditions.

I. I NTRODUCTION 

Nowadays, Large Language Models (LLMs) are extremely large, often requiring a few to hundreds of gigabytes of memory and specialized hardware to operate efficiently. Tra-ditional deployment strategies assume that a single computing device or server can host the entire model [1], which is feasible in cloud environments but impractical for edge or fog computing scenarios [2]. Usage of such computing platforms is increasingly becoming important as they enable low-latency processing and improve data privacy. For example, deploying LLM on edge and fog devices can create accessible, scalable, and personalized mental healthcare ecosystem for anyone with smart mobile devices [3], enable real-time decision-making and contextual understanding on robotic platforms [4], etc. Yet, 

edge devices typically have limited computational resources, making it impossible to store and run a full LLM on a single device . Moreover, due to this constraint, optimizing key performance metrics such as throughput and responsiveness remain a significant challenge when deploying LLMs in edge and fog environments especially in high-demand scenarios. 

Furthermore, many small/medium companies, organiza-tions, and even individual users often need to fine-tune LLMs on their own datasets to meet their specific requirements [5]. These datasets are usually sensitive or confidential, making privacy a critical concern [6]. Renting cloud resources for this purpose is not only expensive but also raises concerns about data security and compliance. For these users, building their own Edge or Fog infrastructures becomes a more practical and cost-effective option. It allows them to keep data local, maintain control over the fine-tuning process, and reduce dependency on external providers, while still benefiting from distributed computing capabilities. Hence, deploying LLMs on Edge/Fog environments is essential to address these chal-lenges. Currently, there are multiple approaches to address ML model distribution on Edge/Fog environment. One of the most common strategies is hierarchical deployment [7], where the size and complexity of the model are adapted according to the level of the device. In this design, lightweight versions of the model run on end-user devices for quick responses, while more capable edge or fog servers handle larger models for intermediate processing. The most resource-intensive tasks are offloaded to the cloud, which hosts the full-scale model. This tiered architecture reduces latency for simple queries, opti-mizes resource usage, and provides scalability. Each version of the model can be obtained using optimization techniques such as quantization, pruning, and knowledge distillation, which reduce memory and computation requirements, and at the same time preserve accuracy as much as possible. However, this method is generally designed based on prediction accuracy, which is much harder to quantify for text generation in LLMs. Another well-known strategy is model parallelism [8], which aims to distribute the computation of large models across multiple resource-constrained devices to leverage all the computation resources. Model parallelism, also called pipeline parallelism, divides the layers of the model across devices, so each device is responsible for a subset of continuous

layers (i.e., stage) [9]–[11]. This approach enables overlapping execution: while one batch is being processed by later stages, the next batch can start at the earlier stages without waiting for the entire model to finish. As a result, pipeline parallelism significantly improves overall system throughput by reduc-ing idle time and increasing resource utilization. However, without careful deployment optimization of the model across computing resource, it introduces pipeline bubbles and stage imbalance: if one stage is slower, all other stages need to wait [9], [10]. Moreover, most state-of-the-art methods leveraging model parallelism assume that all devices in the network participate in a single pipeline, meaning that they collectively host one instance of the model. This assumption introduces two major drawbacks. First, it significantly increases network demand when the number of devices is large, as every stage depends on inter-device communication. Second, it can lead to resource under-utilization: if a subset of devices is powerful enough to host the model efficiently, forcing all devices into one pipeline wastes potential parallelism. Currently, only a few methods involve multiple replicas, such as Splitwise [12] and Thunderserve [1]. However, these methods focus exclusively on cloud deployment, where each device is capable of hosting an entire model or has strong computation power that can finish the task quickly. Hence, there is a need for a replication-based strategy that works in resource-constrained environments like Edge and Fog. In addition, the concept of deploying multiple replicas of the model using model parallelism techniques on heterogeneous edge/fog environment has not been investigated, introducing a potential solution to improve the overall performance of the system. Specifically, such a solution would allow unused devices to form additional clusters, each running a model replica in a distributed fashion. In this paper, we introduce E2LLM, a method for distribut-ing LLMs across heterogeneous Edge/Fog devices using both 

replication and model parallelism . Overall, E2LLM prioritizes latency and bottleneck reduction to ensure responsiveness under resource constraint in Edge/Fog deployment environ-ment. From the pool of available computing devices, we partition them into multiple clusters, with each cluster hosting one replica of the target model. Unfortunately, determining the optimal partitioning and clustering strategy is an NP-hard problem, as the number of possible solutions grows exponentially with the number of devices. To achieve this, we employ a Genetic Algorithm (GA) as a heuristic approach for clustering and resource allocation. Our GA design incorporates a two-chromosome representation to capture both clustering and node ordering, along with an elite strategy that preserves a subset of globally best solutions across generations. This en-sures that high-quality candidates are not lost during evolution. We also differentiate the role of each replica into Prefill and Decoder modes—as illustrated in Figure 1—based on the ratio between input and generated tokens and the computational capability of each device. Moreover, to distribute the full model across devices within a replica, we use Dynamic Programming to minimize commu-nication/computation bottlenecks and determine the optimal configuration, such as the number of concurrent client re-quests, to enhance throughput while still satisfying Quality-of-Service requirements, rather than processing requests sequen-tially. We note that performance guarantees in such systems are commonly expressed in terms of Service Level Objectives (SLOs), including Time-to-First-Token (TTFT) and Time-per-Output-Token (TPOT), rather than using the broader term Quality-of-Service. However, for consistency with established distributed systems terminology, we adopt the term Quality-of-Service to encompass these SLO metrics. The results show that our method achieves up to 2× higher decoding throughput and reduces accumulated waiting time by more than 50% as compared to SplitWise. Deploying LLMs in resource-constrained environments has become an active area of research due to the growing demand for low-latency, privacy-preserving, and cost-efficient AI ser-vices. This challenge involves three main aspects: hierarchical deployment across multiple tiers, parallelization techniques for distributing computation, and replication deployment on heterogeneous devices. 

Hierarchical deployment across multiple tiers. One com-mon approach to handle resource limitations is to organize the deployment in levels, where each tier has a specific role. For example, a powerful cloud server can host the full model, while smaller edge devices run lightweight versions or perform pre-processing. AppealNet [7] adopts a hierarchical deploy-ment strategy for edge/cloud inference, where a lightweight model on the edge handles most inputs, and only hard cases are escalated to a powerful cloud model. This hierarchy is managed by a learnable predictor integrated into the edge model, which decides offloading based on a joint optimization objective that balances accuracy and resource constraints. 

Parallelization techniques for distributing computation. 

To speed up processing and handle large models, researchers use parallelism to split the workload across multiple devices or processors. Pipeline parallelism (or model parallelism) splits the model into stages and assigns each stage to a different Fig. 1. Phases in general LLM Generation Task. 

device. While one stage processes the current input, the next stage can start working on the previous input, creating an assembly-line effect. EdgeShard (Zhang et al.) [10] and Le et al. [9] highlight a key property of pipeline parallelism in edge-cloud inference: while splitting a DNN into multiple stages and executing them concurrently can reduce latency through overlapping computation and communication, the overall throughput is fundamentally limited by the slowest stage in the pipeline. This bottleneck effect means that even if other stages finish quickly, the end-to-end performance depends on balancing partition sizes and resource allocation to avoid idle time. Moreover, graph partitioning tools such as Scotch [13] and METIS [14] have been widely used to improve pipeline par-allelism by balancing computational workloads and minimiz-ing inter-stage communication. These methods aim to create partitions that reduce communication overhead while keeping stages as evenly loaded as possible. However, the fundamental challenge of model parallelism lies in mitigating pipeline bottlenecks while leveraging its key advantage—overlapping computation and communication. Hence, these methods would be inefficient in model parallelism applications. 

Replication deployment. Replication is another strategy to improve reliability and performance in distributed systems. Instead of running a single copy of the model, multiple replicas are deployed across different devices or locations. This approach helps balance the workload because requests can be served by the nearest or least busy replica, reducing latency and increasing throughput. HexGen (Jiang et al.) [15] addresses generative inference of large foundation models in heterogeneous, decentralized cloud environments by introduc-ing replicated pipeline and tensor-parallel execution. Each replica is split into pipeline stages, with tensor-parallel groups assigned to diverse GPUs, and coordinated through leader-based communication. The core optimization problem of Hex-Gen is to minimize the total computation and communication time of pipeline. In 2025, Jiang et al. introduce another method for deploy-ing multiple replication on heterogeneous cloud environment, called ThunderServe. Unlike HexGen, ThunderServe realize that the communication time is the factor that mainly affect the system efficiency. Hence the core optimization problem has been adapted to this observation. However, both methods rely on exhaustive search strategies, so the main optimization objectives become the most critical role in system efficiency. Moreover, ThunderServe differentiates the phases of LLM in-ference into Prefill and Decoder modes, recognizing their dis-tinct computational and communication characteristics. This separation allows ThunderServe to schedule these phases inde-pendently, improving resource utilization and reducing latency under heterogeneous conditions. Unfortunately, both HexGen and ThunderServe ignore the most important characteristic of pipeline parallelism: its ability to overlap computation and communication across stages. This overlap is the factor making pipeline parallelism efficient, as it allows different stages of the model to process data simultaneously, reducing idle time and improving throughput [9], [10]. Splitwise [12] introduces an efficient generative LLM infer-ence framework by leveraging phase splitting, which separates the inference process into Prefill and Decoder phases—as shown in Figure 1—according to ratio of input (prefill) and generated (decode) tokens. As shown in Figure 1, all input tokens are processed during the first iteration (prefill phase), which achieves high throughput by leveraging parallelization on modern GPUs. In contrast, the decode phase generates only one token per iteration, requiring a large number of loops to produce the full output text. This sequential nature signifi-cantly reduces throughput and makes decoding the dominant contributor to overall latency. Consequently, in most cases, the decode phase takes considerably more time to complete compared to the prefill phase. By tailoring parallelism and ex-ecution strategies to each phase, Splitwise significantly reduces energy consumption and latency, and improves throughput compared to uniform execution approaches. However, Split-wise is limited to scenarios where a single device can host the entire model for execution. Moreover, Splitwise makes a manual assumption that the device assigned to the Prefill role must have equal or higher performance compared to those handling the Decoder phase. This constraint reduces flexibility and may lead to inefficient resource utilization in truly heterogeneous environments. Furthermore, HexGen, ThunderServe, and Splitwise pri-marily focus on cloud environments equipped with highly powerful devices that offer substantial computational resources and memory bandwidth. This raises an important question: 

how well can these approaches adapt to edge or fog servers, where resources are far more limited and heterogeneous? De-ploying LLM in such environments introduces new challenges, Fig. 2. Overview of E2LLM Architecture 

including constrained memory, lower compute capacity and memory bandwidth. Addressing these challenges is critical for enabling cost-efficient, low-latency, and privacy-preserving AI services beyond the cloud. III. P ROPOSED METHOD 

Figure 2 illustrates the architecture of our proposed ap-proach, E2LLM. The main objective of our approach is to enable efficient clustering and deployment of LLMs on centralized heterogeneous Edge/Fog environments. To achieve this, we first gather essential information about each node, including its computational capacity and the network band-width between nodes. Using this data, we apply a Genetic Algorithm to identify an optimal clustering strategy that max-imizes throughput while satisfying Quality-of-Service con-straints. Each cluster hosts a replica of the full model, which is then partitioned across devices within the cluster using a Dynamic Programming approach to minimize bottlenecks in pipeline parallelism. This process is supported by latency profiling at the layer level to guide partitioning decisions. Finally, we determine the best number of Prefill and De-coder replicas through brute-force search and assign roles to minimize waiting time under high-demand scenarios. After selecting the best configuration, we deploy the model and use a Join Shortest Queue (JSQ) load balancer [12] to handle high-demand scenarios effectively. 

A. Latency profiling 

To accurately predict system performance, we begin by profiling latency of each layer on each devices. Hence, when having a configuration of distributing a pipeline on a cluster, we can identify the time to finish one inference pass, and the bottleneck within pipeline. However, profiling latency at the layer level is challenging because LLM requires enormous memory, making it impractical to load the entire model for detailed analysis. Fortunately, most LLMs share a common architecture consisting of three main components: a language model head, multiple Transformer blocks—which account for the majority of the model’s memory—and an output layer. To simplify profiling, we assume that Trans-former blocks of the same type (sliding-window attention, full attention, etc.) have similar latency as transformers of the same type have the same number of FLOPs and GPU kernels. Based on this assumption, we adapt llama.cpp —

https://github.com/ggml-org/llama.cpp , which is the famous LLM running framework in C++ language—to load the model with a reduced number of Transformer blocks, allowing us to measure latency efficiently without exhausting system re-sources. It is important to note that in llama.cpp, both the language model head and the output layer must reside on the master device. This constraint ensures correct computation and coordination during inference. 

B. Dynamic Programming method for model partitioning in pipeline parallelism 

As discussed in Section II, pipeline parallelism aims to reduce idle time and minimize bottlenecks across devices. The key challenge lies in partitioning the model so that the slowest stage in the pipeline is minimized. It is worth noting that in this subsection, we focus on partitioning a model across multiple devices in a replica. To address this, we propose a dynamic-programming-based approach to determine an optimal partitioning strategy, guided by latency profiling and memory constraints. The term “dy-namic programming algorithm” here refers to a conceptual framework rather than a fully implemented algorithm. Differ-ent design choices can lead to varying optimization goals and efficiency levels. Moreover, while EdgeShard [10] acknowledges the bottle-neck in pipeline parallelism, their method focuses on finding a deployment plan that maximizes throughput for a single forward pass and then applies pipeline scheduling to reduce idle time and improve overall throughput. This approach overlooks a critical characteristic: when processing multiple batches in pipeline parallelism, the slowest stage becomes the bottleneck, as demonstrated in [9]. Our experiments confirm this behavior. If we have M

devices in a pipeline and Tslowest represents the latency of the slowest stage, then the TPOT (or time to finish one forward pass) is approximately Tslowest × M . However, the average TPOT per request remains Tslowest since M batches are processed concurrently. Additionally, since we are distributing multiple Transformer blocks of identical size, the ordering of nodes in the pipeline is generally irrelevant, except for the node assigned the master role. This master node is responsible for hosting the language model head and output layer. Hence, we just use random or sorted order of the node. For the master node, we simply add the latency of the language model head and the output layer in the dynamic programming process. D. Genetic Algorithm for deployment LLM on Heterogeneous Edge/Fog Environment 

Designing an optimal deployment plan for LLM across heterogeneous Edge/Fog devices is a complex combinatorial problem. The search space grows exponentially with the number of devices, model layers, and constraints such as memory, latency, and communication overhead. Traditional exhaustive search methods become impractical in such sce-narios due to their computational cost. To tackle this chal-lenge, we adopt a Genetic Algorithm (GA) approach, as shown in Algorithm 2. GA is well-suited for problems with large, irregular search spaces because it uses evolutionary principles—selection, crossover, and mutation—to efficiently explore candidate solutions and converge toward near-optimal deployment strategies. This enables us to balance performance objectives, such as minimizing bottlenecks and maximizing throughput, without incurring prohibitive computation time.

However, GA is only a tool; its effectiveness depends entirely on how the core logic is implemented. Critical design choices—such as gene representation, mutation strategy, and crossover policy—play the most important role in determining the quality of solutions and convergence speed. A well-designed GA tailored to the characteristics of LLM deploy-ment can significantly outperform generic implementations. 

Gene definition. In this work, we represent each gene using two chromosomes. The first chromosome encodes the ordering of nodes/devices, while the second chromosome specifies the grouping of nodes into replicas. For example, the gene: 

[(0 , 1, 2, 3, 4, 5) , (3 , 1, 2)] indicates that there are three replicas. The first replica contains nodes 0, 1, and 2; the second replica consists of node 3; and the remaining nodes (4 and 5) form the third replica. This representation allows the Genetic Algorithm to explore both device ordering and replica composition during optimization. 

Cross-over policy. During crossover, we randomly select two candidate genes based on their fitness scores (effec-tiveness). The first chromosome, which represents the node ordering, is crossed over between the two parents, while the second chromosome, representing replica grouping, is randomly chosen from one of them. However, this process can produce invalid genes because node duplication may occur in the first chromosome. To address this, a repairing procedure is applied to ensure that each node appears exactly once, restoring the validity of the gene before proceeding to the next generation. 

Mutation strategy. Mutation introduces diversity into the population and prevents premature convergence. In our ap-proach, mutation operates on both chromosomes of a gene. We set the probability of a gene be mutated to about 30%. When a gene is decided to be mutated, there are:  

> •

20% chance that we exchange two random position in the first chromosome.  

> •

50% chance that we destroy all replica size from a random position in the second chromosome and then regenerate again.  

> •

15% chance that we completely destroy the second chro-mosome and regenerate again.  

> •

15% chance that we destroy both chromosome and re-generate again. 

Elite citizen. In addition to traditional crossover and mu-tation, we implement an elite preservation strategy to main-tain high-quality solutions across generations. Specifically, we define a maximum number of globally best-performing individuals (elite citizens) to retain after each iteration. These elite individuals are then used in crossover operations with the current population, ensuring that strong genetic material propagates through subsequent generations. This strategy ac-celerates convergence toward optimal deployment plans while reducing the risk of losing high-quality solutions during ran-dom mutations. After generating a chromosome (line 1 and 20-22), the next step is straightforward. We decompose the chromosome into multiple replicas according to its grouping structure and then apply Algorithm 1 to determine the optimal deployment plan for each role. For Prefill role, most of the time, only one request can fill up maximum tokens to be processed at once (line 7-8). For Decoder role, we need to find the highest number of parallel requests (line 9-15) that can be processed at the same time to increase throughput while guaranteeing the minimum generated token per second. Moreover, we also cache the result of each replica for reuse, decreasing processing time. This integration ensures that the Genetic Algorithm leverages dynamic programming for fine-grained optimization within each replica, combining global search with local refinement. Finally, based on the analysis in the previous subsection, we apply a brute-force method to assign the role of each replica in each candidate with the objective of minimizing the bottleneck, as defined in Equation 4 (line 16). This step ensures that the deployment plan not only respects the grouping structure provided by the Genetic Algorithm but also optimizes role assignment for maximum efficiency. In addition, because the number of replicas in Edge/Fog envi-ronments is typically small, applying a brute-force approach for role assignment remains feasible and maximum number of parallel requests. This allows us to exhaustively evaluate all possible configurations to minimize the bottleneck objective, ensuring optimal deployment without incurring prohibitive computational overhead. 

E. Quality-of-Service Definition 

In Edge and Fog LLM deployment, the system often faces heavy demand. Multiple requests may need to wait in a queue before processing begins, making it difficult to set a strict threshold for waiting time. However, once processing starts, the speed of text generation becomes critical. This generation rate directly shapes the user experience—slow output can feel frustrating and break the natural flow of interaction. Therefore, we focus on defining Quality-of-Service in terms of text generation speed, as it is the most immediate and noticeable factor for users. Brysbaert et al. [19] reports that the average human reading speed is about 238 words per minutes, roughly four words per second. However, some users read much faster, especially when skimming through text. To account for these cases, we assume that fast readers can process between 7 and 15 words per second. For simplicity, we also treat each word as equivalent to one token in typical conversational text. Based on this assumption, the system should generate text at a minimum rate of 7 tokens per second , with higher speeds providing an even better user experience. IV. E VALUATION AND DISCUSSION

Through this evaluation, we aim to demonstrate how E2LLM improves responsiveness and scalability compared to existing approaches under resource-constrained conditions.

Another key factor that affects system performance is network bandwidth, especially during the decoder phase.
