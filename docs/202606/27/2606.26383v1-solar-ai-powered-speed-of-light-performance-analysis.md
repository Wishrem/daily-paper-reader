## 摘要
深度学习模型在目标硬件上能运行多快？当前实现离这一极限还有多远？这些问题对软件、硬件和算法优化至关重要。光速（SOL）分析通过计算工作负载在给定架构上的理论最小执行时间来回答这些问题。然而，推导SOL边界仍然依赖人工、容易出错，并且与快速模型开发脱节。为弥补这一差距，我们提出了SOLAR，一个能够从PyTorch和JAX源代码自动推导已验证SOL边界的框架。SOLAR在其流程中结合了生成式组件和确定性组件：一个LLM前端将任意源程序转换为可执行的仿射循环IR（中间表示），并通过输出比较进行验证；一个确定性流程将IR提升为einsum图；一个分析后端计算未融合、融合和缓存感知的SOL边界。SOLAR提供全面的算子和语言覆盖，生成零违反记录的已验证边界，并提供多保真度分析，从而收紧边界并揭示优化洞见。我们在KernelBench、JAX/Flax模型和机器人工作负载上评估了SOLAR。这些实验展示了四个用例：多保真度级别的余量分析、识别优化机会、跨平台探索以及逆向屋顶线硬件配置。

## Abstract
How fast could a deep-learning model run on target hardware, and how far is today's implementation from that limit? These questions are central to software, hardware, and algorithm optimizations. Speed-of-Light (SOL) analysis answers them by computing a workload's theoretical minimum execution time on a given architecture. Yet deriving SOL bounds remains manual, error-prone, and disconnected from rapid model development. To close this gap, we introduce SOLAR, a framework that automatically derives validated SOL bounds from PyTorch and JAX source code. SOLAR leverages both generative and deterministic components in its flow: an LLM frontend translates any source programs into an executable Affine Loop IR, validated by output comparison; a deterministic flow lifts the IR into an einsum graph; and an analytical backend computes unfused, fused, and cache-aware SOL bounds. SOLAR provides comprehensive operator and language coverage, produces validated bounds with zero observed SOL violations, and offers multi-fidelity analysis that tightens bounds and surfaces optimization insights. We evaluate SOLAR across KernelBench, JAX/Flax models, and robotics workloads. These experiments demonstrate four use cases: headroom analysis at multiple fidelity levels, identifying optimization opportunities, cross-platform exploration, and inverse-roofline hardware provisioning.

- **背景**：现有工具（如FLOP计数器 `fvcore`、`ptflops`）仅覆盖75‑84%的算子且忽略内存流量；硬件分析器（NCU、NSys）测量的是**已实现**性能而非理论极限；纯LLM估计虽然直接，但准确率不高（复合场景仅38%），且无法处理系统性的搜索优化（如缓存感知分块）。因此，亟需一个**自动、可验证、多保真度**的SOL边界推导框架。

## 2. 方法论：核心思想、关键技术细节

### 2.1 整体架构：三阶段流水线（生成+确定性分离）

SOLAR将LLM的翻译能力与确定性分析相结合，分为三个串行阶段：

1. **验证性智能体前端**（Validated Agentic Frontend）
   - **输入**：任意PyTorch或JAX源码程序。
   - **中间表示**：生成一个可执行的**仿射循环IR**（Affine Loop IR），它由两部分组成：
     - **仿射循环内核**：每个内核是严格嵌套的循环，循环边界为静态张量形状，索引表达式为仿射形式，无数据依赖分支，体部只含标量运算和固定内置函数。这便于后续机械式地提取einsum。
     - **无限制的组合层**：用Python函数组合内核、调用库内置函数、实现控制流（如reshape、concat、gather等）。
   - **关键流程**：LLM通过“生成‑验证“循环完成翻译：
     1. 先运行原始程序获取**Golden输出**。
     2. LLM生成IR代码（Python+ Numba加速）。
     3. 执行IR并与Golden输出对比，若不匹配则提供诊断反馈给LLM修正。
     4. 最终只保留通过数值验证的IR。
   - **语言可移植性**：只需添加源语言适配器（模型捕获模块+语言约定提示），新语言即可支持。

2. **确定性Einsum生成**（Deterministic Einsum Generation）
   - 从已验证的IR中机械地提取**Einsum图**：一个有向无环图（DAG），节点为扩展einsum（可包含max‑reduction等），边为中间张量。
   - **Einsum提取规则**：给定一个`@kernel`，其**输出维度**为DO，**循环索引**为D，则归约维度DR = D \ DO。每个输入张量Ti产生一个einsum操作数Di→DO，隐式求和在DR上。
   - **自动推导计算和内存量**：
     - MACs = ∏_{d∈DO∪DR} |d|
     - Memory traffic = ∑_i sizeof(Ti)·∏|Di| + sizeof(Tout)·∏|DO|
   - 该阶段还会检测非einsum节点（如gather、reshape）并标记为纯数据移动（MACs=0）。

3. **SOL分析**（SOL Analysis）
   - **基本屋顶线模型**：对于给定硬件峰值吞吐Π (FLOP/s) 和带宽β (B/s)：
     T_SOL = max(FLOPs/Π, Traffic/β)
   - **多保真度模式**（三者共享相同FLOPs，仅流量统计不同）：
     - **未融合**（Unfused）：各算子独立，中间张量都从DRAM读写。
     - **融合**（Fused）：相邻算子共享片上级缓存，消除中间张量的DRAM往返。
     - **缓存感知融合**（Cache‑aware, OROJENESIS）：考虑片上缓存容量限制，当工作集过大时必须分块导致元素被重新读取，通过整数线性规划求解最小化数据移动。
   - **逆屋顶线**：给定目标延迟T_target，反推所需的最小Π_min和β_min，用于硬件配置。

### 2.2 关键技术细节补充

- 仿射循环IR的四大设计目标：1) 易于机械提取einsum；2) 表达力覆盖各类DL工作负载；3) 易于LLM生成（LLM已擅长Python）；4) 可执行以进行数值验证。
- Einsum图支持**融合性分析**：相邻操作间若共享中间张量，则可消除该张量的DRAM流量。
- 缓存感知分析依赖于AccelForge框架和整数线性规划求解数据移动，比朴素屋顶线更紧。

## 6. 主要结论与发现

1. **自动化SOL边界推导可行且有效**：SOLAR是首个从源码自动生成已验证SOL边界的工具，在KernelBench上达到100%覆盖率，且零SOL违反。
2. **巨大优化余量普遍存在**：KernelBench L3的eager模式与融合SOL之间余量可达54.6x几何平均；torch.compile仅部分缩小该差距（L3仍有47.7x）。
3. **融合和缓存感知显著收紧边界**：L1上缓存感知SOL比朴素屋顶线紧2.06x；L2上融合消除中间张量后余量从5.1x增至21.6x。
4. **Einsum重排序可大幅降低计算量**：DeepSeek MLA的最优结合顺序比朴素左到右减少2.04x FLOP，范围达22x。
5. **跨平台投影和逆屋顶线提供硬件规划指导**：机器人模型在Jetson Thor上全部内存受限；500Hz伺服控制需要6.4‑19.7x当前带宽，表明需结合模型压缩和工艺改进。
6. **语言无关性验证**：JAX/Flax覆盖8个模型，体现框架可扩展性。

## 7. 优点

- **自动且可验证**：通过LLM翻译+数值验证，免去手动建模，且翻译正确性有保证。
- **多保真度分析**：未融合/融合/缓存感知，满足不同粒度优化需求。
- **语言无关**：易于扩展到新框架（只需适配器）。
- **关注理论极限而非已实现性能**：比分析器更适合识别天花板。
- **开源**：代码已发布（https://github.com/NVlabs/SOLAR），便于复现和扩展。
- **输出标准化Einsum图**：可与Timeloop等下游建模框架直接对接。
- **反向问题（逆向屋顶线）**：对硬件架构师特别有用，可在芯片制造前评估需求。

## 8. 不足与局限

- **省略值相关优化**：SOLAR基于张量形状分析，无法捕获因数据值特性（如稀疏模式、常数传播、动态形状）带来的性能提升。
- **硬件可变性未建模**：SOL边界假设理想重叠和峰值性能，实际可能因电源上限、热节流、内核启动开销等无法达到，导致边界偏乐观。
- **LLM翻译无形式保证**：即使通过验证，仍可能因测试输入未覆盖所有路径而隐藏错误。
- **缓存模型有限**：仅考虑一级片上缓存，未建模多级缓存层次和带宽共享效应。
- **仅覆盖两框架（PyTorch和JAX）**：若需支持TensorFlow、MindSpore等需额外适配工作。
- **缺乏与其他自动SOL工具的直接对比**：由于SOLAR是首个此类工具，论文只能对比部分功能的子集（FLOP计数器、LLM估计），未与同类全生命周期工具比较。
- **未提供多保真度之间的计算开销分析**：缓存感知求解（整数线性规划）可能较慢，但论文未给出典型运行时间。
